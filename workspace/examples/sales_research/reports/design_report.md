## Phase 1: ナレッジベース構造

### 設計方針

assessの推奨手段は **エージェント** のため、種類別に複数のナレッジベースを設計する。

```
エージェント向けナレッジ設計:
  ├── KB-1: 製品FAQ（事実系）← 自社製品の機能・価格・強み弱みへの直接回答
  ├── KB-2: 競合情報KB（比較系）← 競合との比較ポイント・勝ち負けの場面
  ├── KB-3: 営業手順・ポリシー（ルール系）← 提案手順・反論対応・値引き基準・エスカレーション
  └── KB-4: 過去事例KB（事例系）← 業種別の勝因・敗因パターン
```

### 各ナレッジベースの仕様

#### KB-1: 製品FAQ

| 項目 | 内容 |
|------|------|
| ナレッジベース名 | 製品FAQ |
| 目的 | 自社5製品の機能・価格・強み・弱みへの直接回答。「TechFlow CRM Proの価格は？」「TechFlow MAの強みは？」 |
| 推定項目数 | 40〜50件 |
| ソースドキュメント | data/product_catalog.md |
| チャンキング戦略 | 1FAQ = 1チャンク（128〜512トークン）。製品×質問タイプ別に分割 |
| メタデータ | product（製品名）, category（機能/価格/強み/弱み/対象顧客）, last_updated |

**生成するFAQの観点**:
- 各製品の主要機能（3〜5件/製品）
- 各製品の価格（プラン別）
- 各製品の強み・弱み
- 各製品の対象顧客
- セット割引・製品間の連携
- 競合との直接比較（製品レベル）

#### KB-2: 競合情報KB

| 項目 | 内容 |
|------|------|
| ナレッジベース名 | 競合情報KB |
| 目的 | 競合3社（Salesforce・HubSpot・GENIEE）との比較情報。「Salesforceと比べてどうか」「HubSpotに負ける場面は」 |
| 推定項目数 | 25〜35件 |
| ソースドキュメント | data/competitor_info.md |
| チャンキング戦略 | 1FAQ = 1チャンク（128〜512トークン）。競合×比較軸別に分割 |
| メタデータ | competitor（競合名）, comparison_axis（価格/機能/サポート等）, win_lose（TechFlowが勝てる場面/負ける場面）, last_updated |

**生成するFAQの観点**:
- 各競合の概要・市場ポジション
- 競合ごとの価格比較
- 各競合にTechFlowが勝てる場面
- 各競合にTechFlowが負ける場面
- よく聞かれる競合質問と回答ポイント

#### KB-3: 営業手順・ポリシーKB

| 項目 | 内容 |
|------|------|
| ナレッジベース名 | 営業手順・ポリシーKB |
| 目的 | 提案手順・反論対応・値引き基準・エスカレーション基準などのルール・手順情報 |
| 推定項目数 | 30〜40件 |
| ソースドキュメント | data/sales_playbook.md |
| チャンキング戦略 | 章・セクション単位でチャンク（256〜1024トークン）。手順書はステップ単位 |
| メタデータ | category（提案手順/反論対応/値引きルール/エスカレーション基準）, target_role（AE/SDR/マネージャー）, last_updated |

**生成するFAQの観点**:
- 提案の基本フレーム「3つのWHY」
- 業種別の刺さるメッセージ
- よくある反論と対処法（反論別）
- 見積もりルール・構成
- 値引き権限の基準（役職別）
- エスカレーション基準（SE・上長・法務等）

#### KB-4: 過去事例KB

| 項目 | 内容 |
|------|------|
| ナレッジベース名 | 過去事例KB |
| 目的 | 業種・状況が似た過去の受注・失注事例の参照。「製造業への提案で過去に何が勝因だったか」 |
| 推定項目数 | 20〜30件（受注5件 + 失注5件 × 3〜4チャンク/件） |
| ソースドキュメント | data/past_deals.md |
| チャンキング戦略 | 1事例 = 2〜3チャンク（背景/勝因分析/学び）。1チャンク512〜1024トークン |
| メタデータ | deal_result（受注/失注）, industry（業種）, company_size（規模）, competitor（競合）, key_factor（主要な勝因/敗因）, last_updated |

**生成するFAQの観点**:
- 業種別の勝因パターン
- 業種別の失注リスク
- 競合別の対戦実績
- 勝因・敗因サマリー

## Phase 2: ナレッジの投入方式

### 2.0 FAQ化するか、そのまま投入するか

**判定結果: 混合（FAQ化 + 一部そのまま）**

| ソース | 判定 | 理由 |
|--------|------|------|
| product_catalog.md | FAQ化 | 製品情報はQ&A形式で答えることが多い。「CRM Proの価格は？」「MAの強みは？」に直接答えるFAQが必要 |
| competitor_info.md | FAQ化 | 競合比較も「Salesforceと比べてどうか？」という形でFAQが有効 |
| sales_playbook.md | 混合 | 反論対応・提案フレームはFAQ化。値引き基準・エスカレーション基準はポリシー文書としてそのまま投入 |
| past_deals.md | そのまま（チャンク分割） | 事例は物語形式で文脈が重要。FAQ化より事例単位のチャンクで投入する方が検索精度が高い |

### 2.1 FAQ生成パターンの選択

**選択パターン: パターンB（2ステップ生成）**

理由:
- ドキュメントが長く・情報が密度高い（製品カタログ・プレイブック）
- 1ステップで一気に生成すると質問の網羅性が低くなるリスクがある
- まず「どんな質問が想定されるか」をリストアップし、次に回答を生成する

```
ステップ1: 質問リストの生成
  → 各ドキュメントから「営業担当者が聞きそうな質問」を列挙
  → 重複排除・カテゴリ分け

ステップ2: 回答の生成
  → ステップ1の質問リストに対して、ドキュメントから回答を生成
  → FAQ品質基準に従って整形
```

### 2.2 FAQ生成の具体的仕様

| 項目 | 内容 |
|------|------|
| 生成対象ドキュメント | product_catalog.md（KB-1用）、competitor_info.md（KB-2用）、sales_playbook.md（KB-3用） |
| 使用パターン | B（2ステップ生成） |
| カテゴリ設定 | 製品情報 / 競合比較 / 提案手順 / 反論対応 / 値引きルール / エスカレーション |
| 推定FAQ件数 | 約100件（3KB合計） |
| 出力形式 | CSV（knowledge/faq_draft.csv） |
| レビュー担当 | 営業チームリーダー（特に反論対応・値引きルールは要確認） |

### 2.3 FAQ品質基準

生成したFAQが満たすべき基準:

- [x] 1FAQ1テーマ
- [x] 結論先出し（最初の1〜2文で回答）
- [x] 条件が明示されている（例: 「10%値引きは年払い選択時に適用」）
- [x] 例外が記載されている（例: 「ただし、Enterprise案件は別途上長承認」）
- [x] 次の行動が書かれている（例: 「詳細は営業マネージャーに確認してください」）
- [x] 参照元が記載されている（例: 「source: sales_playbook.md 第4章」）
- [x] 言い換え・別名を含む（例: 「Salesforce, セールスフォース, SFDC」）

## Phase 3: チャンキング設定

| 文書タイプ | 分割単位 | Chunk Size | Overlap | 備考 |
|-----------|---------|-----------|---------|------|
| FAQ文書（KB-1, KB-2, KB-3） | 1FAQ = 1チャンク | 500 | 0 | Q&Aが自己完結している |
| 過去事例（KB-4） | 事例の構成要素（背景/勝因/学び）単位 | 800 | 100 | 文脈を保持するためOverlapを設定 |
| ポリシー文書（値引き基準等） | 条件・ルール単位 | 500 | 50 | 条件の前後関係が重要 |

### 前処理の必要性

- [x] Q&A形式への変換が必要（product_catalog.md、competitor_info.md、sales_playbook.md）
- [ ] 個人情報の除去（社名はアルファベット表記済み）
- [x] 表記ゆれの統一（「セールスフォース」「Salesforce」「SFDC」等の統一）
- [ ] セクション単位への事前分割（構造化済みのため不要）
- [x] メタデータの付与（各チャンクにcategory, source, last_updated等を付与）

## Phase 5: プロンプト設計

### システムプロンプト

```
あなたはTechFlow株式会社の営業チーム専用の「競合調査・提案準備アシスタント」です。

## 役割
営業担当者が顧客への提案を準備する際に、以下を支援します:
- 自社製品の強み・弱みの整理
- 競合との差別化ポイントの特定
- 過去の類似事例の参照
- 反論対応・提案手順のアドバイス

## 利用可能なツール
1. 製品FAQ検索: 自社製品の機能・価格・強み・弱みを検索
2. 競合情報検索: 競合3社（Salesforce・HubSpot・GENIEE）との比較情報を検索
3. 営業手順・ポリシー検索: 提案手順・反論対応・値引き基準・エスカレーション基準を検索
4. 過去事例検索: 業種別・競合別の過去の受注・失注事例を検索
5. Web検索: 最新の競合情報・顧客企業の最新ニュースを検索（ナレッジに情報がない場合）

## 手順
1. ユーザーの質問を理解し、どんな情報が必要かを判断する
2. 必要なツールを選択し、情報を収集する（複数ツールを組み合わせて良い）
3. 収集した情報を統合し、営業担当者が即座に使えるフォーマットで回答する

## 回答フォーマット
- 提案準備の質問: 「自社の強み」「競合の弱み」「推奨する訴求ポイント」「想定される反論と対処法」の4セクションで整理
- 競合比較の質問: 比較表または箇条書きで整理
- 過去事例の質問: 事例の概要・勝因/敗因・今回への示唆の3点セットで回答

## 禁止事項
- ナレッジに記載されていない値引き率や特別条件を確約しない
- 競合他社を根拠なく誹謗中傷しない
- 未公開のロードマップ情報・内部情報を回答しない
- 法的なアドバイス・契約の解釈について断言しない

## フォールバック
- ナレッジに情報が見つからない場合: 「この件については営業マネージャーまたは担当部署に確認してください」と回答
- 禁止事項に該当する場合: 「この内容についてはお答えできません。XX部門にご確認ください」と回答
```

### ツール定義（エージェント用）

```yaml
tools:
  - name: search_product_faq
    description: "TechFlow製品（CRM Pro/MA/CS/Analytics/Connect）の機能・価格・強み・弱みを検索する"
    when_to_use: "製品の詳細を知りたい時、価格を確認したい時、自社製品の強みを整理したい時"
    knowledge_base: "製品FAQ"

  - name: search_competitor_info
    description: "Salesforce・HubSpot・GENIEEとの比較情報を検索する"
    when_to_use: "競合との違いを説明したい時、競合に勝てる場面・負ける場面を知りたい時"
    knowledge_base: "競合情報KB"

  - name: search_sales_playbook
    description: "提案手順・反論対応・値引き基準・エスカレーション基準を検索する"
    when_to_use: "提案書を作りたい時、反論への対処法を確認したい時、値引きや上長承認の基準を確認したい時"
    knowledge_base: "営業手順・ポリシーKB"

  - name: search_past_deals
    description: "業種・競合・状況が似た過去の受注・失注事例を検索する"
    when_to_use: "同業種への提案で過去にどう勝ったか/負けたかを確認したい時、勝ちパターンを参考にしたい時"
    knowledge_base: "過去事例KB"

  - name: web_search
    description: "最新の競合情報・顧客企業のニュース・業界トレンドをWeb検索する"
    when_to_use: "社内ナレッジに情報がない時、最新情報（直近の競合の動向・顧客企業のニュース）が必要な時"
    provider: "Bing Search API"
```

## Phase 6: ルーティング設計

エージェント型のため、Question Classifierによる固定ルーティングは不要。エージェントが自律的にツールを選択する。

ただし、エージェントが効率的にツールを選択できるよう、以下の**ツール選択ヒント**をシステムプロンプトに組み込む:

```
質問タイプ別の推奨ツール選択:
- 「〇〇製品の〜は？」→ search_product_faq
- 「Salesforceと比べて〜」→ search_competitor_info
- 「〜への反論対応は？」→ search_sales_playbook
- 「〜業種での事例は？」→ search_past_deals
- 「〇〇社について〜」→ web_search（顧客ニュース収集）
- 「A社への提案を整理して」→ 全ツールを順番に使う
```

## 次のステップ

1. `knowledge/faq_draft.csv` の生成（パターンBで各ナレッジドキュメントからFAQを生成）
2. 営業チームリーダーによるFAQレビュー（特に反論対応・値引きルールの確認）
3. Dify Self-hostedへの4つのナレッジベースの作成・データ投入
4. エージェントアプリの構築（システムプロンプト・ツール設定）
5. テスト質問セット（20〜30件）を使った評価（`/kb-eval`）

→ `/kb-build` でDifyアプリの構築手順を生成
