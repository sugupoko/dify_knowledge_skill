## Phase 1: ドキュメントの理解（何があるか）

### 1.1 既存FAQの有無と品質

**既存FAQは存在しない。** 前提条件通り、マニュアルはあるがFAQ一覧は未配布・未整備。

問い合わせログの品質チェック（FAQ変換前提として評価）:

| チェック項目 | 評価 |
|-------------|------|
| 1FAQ1テーマになっているか | 問い合わせの多くは1テーマ。ただし一部複合（例: 問い合わせ038「PC・アカウント・メール・共有フォルダ全部まとめて」） |
| 結論が先に来ているか | 該当なし（質問のみで回答なし） |
| 条件・例外が明示されているか | 該当なし（回答なし） |
| 言い換え・別名があるか | 「有休」「有給」などの俗称は一部あるが体系的ではない |
| 参照元が記載されているか | 該当なし |

**判定: マニュアルから新規FAQ生成が必要。問い合わせログは「よくある質問のトピック抽出」に活用する。**

### 1.2 マニュアルの特性

マニュアルは手順書とポリシーが混在した構成。以下の分類が必要:

| 情報の種類 | 例 | 変換先 |
|-----------|-----|--------|
| 手順（5ステップ以上） | PCセットアップ手順、パスワードリセット手順 | 層2（手順書）としてチャンク化 |
| 短い手順（3〜4ステップ以下） | プリンター紙詰まり処理 | 層1（FAQ）に変換可能 |
| ポリシー・ルール | パスワードポリシー、データ持ち出し禁止 | 層2（規程・ポリシー）+ FAQ化 |
| 連絡先・申請先 | ヘルプデスク内線9999、社内ポータルのURL | 層1（FAQ）に含める |
| 緊急対応手順 | ランサムウェア対応、インシデント報告 | 層3（エスカレーションルール）+ FAQ |

## Phase 2: 目標の理解（何を作るか）

### 2.1 ヒアリング結果の整理

| 項目 | 内容 |
|------|------|
| **目的** | 問い合わせ対応の自動化（繰り返し問い合わせの削減） |
| **対象ユーザー** | 社内従業員 約500名（非エンジニアが大半） |
| **問い合わせ件数** | 月約200件（現在3名体制で対応） |
| **課題** | 同じ質問の繰り返し / 3名では対応しきれない / 夜間・休日対応不可 |
| **成功基準** | 問い合わせの50%を自動化（月100件の削減） |
| **Dify環境** | Self-hosted（オンプレ）/ Ollama（ローカルLLM） |
| **制約** | FAQは未整備。マニュアルのみ存在 |

### 2.2 ユーザーの状態（軸1）

**判定: 「聞きたいことは分かっているが、キーワードや正式名称を知らない」**

根拠:
- 問い合わせログを見ると、ユーザーは「VPNが繋がらない」「パスワードを忘れた」など自分の困りごとを認識している
- しかし「GlobalProtect」「スプリットトンネル」「セルフサービスポータル」などの正式名称や技術用語は知らないケースが多い
- 「なんかPCの調子が悪い」のような曖昧な表現も10件中2件（4%）存在し、会話で絞り込むことが有効
- 非エンジニアが大半のため、キーワード検索より自然な質問形式の方が使いやすい

### 2.3 情報の構造（軸2）

**判定: 「条件で答えが分岐する」かつ「1箇所で完結する」が混在**

根拠:
- **1箇所で完結する質問（多数）**: パスワードリセット方法、申請先の確認、プリンター操作など
- **条件で答えが分岐する質問（一定数あり）**: 退職アカウントの扱い（最終出社日 vs 退職日の違い）、ソフトウェア申請（許可リスト掲載済み vs 未掲載）、バッテリー交換（2年経過 + 著しい劣化の両条件）など
- **緊急度による分岐**: ランサムウェア感染、共有フォルダ消失などは通常FAQとは異なる即座のエスカレーションが必要
- **複数情報源が不要**: ほとんどの質問はマニュアル内の1章で回答できる（複数チャプターを横断する必要がない）

### 2.4 推奨手段の選定

**推奨: チャットボット（Dify チャットボット + ナレッジベース）**

**選定の根拠:**

app_type_selection_guide.md のマトリクスより:
- ユーザー状態:「聞きたいことは分かっている」
- 情報の構造: 「1箇所で完結」が中心（条件分岐は一部）
→ **チャットボット**が最適

**数値目安との照合:**

| 指標 | 基準値 | 本プロジェクト | 判定 |
|------|--------|---------------|------|
| FAQ件数 | 100件以下 | 推定60〜80件 | チャットボット向き |
| ナレッジベース数 | 1〜2つ | 2〜3つを想定 | チャットボット向き |
| 条件分岐の複雑さ | 低〜中 | 低（1段階程度） | チャットボット向き |

**チャットボットが強い要因（本プロジェクト固有）:**

1. **24時間・夜間対応のニーズ**: 現在夜間・休日対応不可。これがチャットボットの最大の価値
2. **非エンジニアが大半**: 検索型より自然な会話が使いやすい
3. **FAQ未整備**: 既存FAQ一覧がないため「チャットボットで新規提供する価値がある」
4. **Ollama（ローカルLLM）使用**: エージェントは多段推論のコストが高く、ローカルLLMには不向き

**チャットフローを選ばない理由:**

問い合わせは50件中、本質的に条件分岐が必要なものは10件程度。大半は「この場合はどうする？」ではなく「これはどこで申請するの？」「手順を教えて」タイプ。チャットフローを導入すると構築・運用コストが上がるが、得られる精度向上は限定的。**まずチャットボットでPoC → 解決できないものがあればチャットフローに拡張**の段階的アプローチを推奨。

**代替案: チャットフロー**

理由: カテゴリが明確（VPN、アカウント、PC、セキュリティ等）なため、Question Classifierでルーティングすれば精度向上の余地がある。ただし構築コストが上がるため、チャットボットで解決率が不十分な場合（目標50%未達）の次フェーズとして推奨。

**エージェントを選ばない理由:**

Ollama（ローカルLLM）環境での多段推論は精度が不安定になりやすく、ランニングコストも高い。本プロジェクトの質問は「複数情報源を組み合わせる必要がない」ため、エージェントは過剰。

### 2.5 必要なナレッジ種別の特定

#### 層1: FAQ（直接回答層）― 必須

- **推定件数**: 60〜80件
- **ソース内訳**:
  - マニュアルから変換: 50〜60件（各章の主要な手順・ポリシーをFAQ化）
  - 問い合わせログから追加: 10〜20件（ログに出てきてマニュアルにない情報）
- **優先度高カテゴリ**（問い合わせログの件数が多い順）:
  1. PC・ハードウェア（6件のトピック）
  2. メール・Teams（6件）
  3. パスワード・アカウント（5件）
  4. VPN・ネットワーク（5件）
  5. セキュリティ（5件）

#### 層2: 支援文書（詳細・手順層）― 必要

| 種類 | 要否 | 理由 |
|------|------|------|
| 手順書 | 必要 | PCセットアップ（7ステップ）、パスワードリセット（5ステップ）など5ステップ超の手順あり |
| 規程・ポリシー | 必要 | パスワードポリシー、データ持ち出しルール、BYOD規程など |
| 用語集 | 必要（優先度: 中） | GlobalProtect、スプリットトンネル、MDM等の技術用語が非エンジニアには不明 |

#### 層3: 運用情報（メタ層）― 必要

| 種類 | 要否 | 理由 |
|------|------|------|
| エスカレーションルール | **必須** | ランサムウェア・情報漏洩などのインシデントはボットが処理すべきでない。即座に人間へ繋ぐルールが必要 |
| 更新管理 | 必要 | マニュアルの更新日が不明。古い情報で回答するリスクあり |
| 禁止事項 | 必要 | 個人アカウント情報、パスワード直接聞き出し、法的判断等は回答禁止にする必要あり |

## Phase 4: 推奨設計の概要

### ユーザーの状態

- **ユーザーは何をどこまで分かっている状態で来るか**: 聞きたいことは分かっているが、キーワード・正式名称を知らない
- **根拠**: 問い合わせログの表現（「VPN繋がらない」「パスワード忘れた」等）は日常語で、技術用語（GlobalProtect、スプリットトンネル等）は使われていない。非エンジニア大半の環境では、会話形式の方が検索型より使いやすい。

### 情報の構造

- **答えはどこにあるか**: 主に1箇所で完結（一部条件分岐あり）
- **根拠**: 問い合わせログ50件のうち約40件はマニュアルの特定の章・節で回答できる。残り10件程度は条件分岐や緊急対応が必要だが、フロー制御ではなくエスカレーション設計で対応可能。

### 推奨手段

- **推奨**: **チャットボット**（Dify チャットボット + ナレッジベース）
- **理由**: 「聞きたいことは分かっているが、キーワードを知らない」× 「1箇所で完結」の組み合わせ。FAQ未整備・夜間対応ニーズ・非エンジニアユーザーの3条件がチャットボットの価値を最大化する。ローカルLLM（Ollama）環境では、エージェントより予測可能なチャットボットの方がコスト・安定性の観点で有利。
- **代替案**: チャットフロー。解決率が50%に届かない場合、カテゴリ別ルーティングを追加することで精度向上が見込める。チャットボットで初期PoC後に段階的に移行することを推奨。

### 必要なナレッジ

- **層1（FAQ）**: 推定60〜80件。ソース: マニュアルから変換（50〜60件）+ 問い合わせログからの補完（10〜20件）
- **層2（支援文書）**: 手順書（PCセットアップ、パスワードリセット等）、規程・ポリシー（パスワードポリシー、データ持ち出し、BYOD）、用語集（IT用語の別名・略語辞書）
- **層3（運用情報）**: エスカレーションルール（必須：インシデント系の即時エスカレーション）、禁止事項（個人情報・法的判断等）、更新管理

### 仮定

- FAQは60〜80件で月100件（50%）の問い合わせに対応できると仮定（根拠: マニュアルカバレッジ約70%、重複問い合わせの存在）
- 問い合わせの約70%はマニュアルで回答可能と仮定（根拠: ログ目視確認）
- Ollamaモデルは日本語対応の7B〜13Bクラスと仮定
- 夜間・休日の問い合わせは全体の20〜30%と仮定

→ **これらの仮定が正しいか確認をお願いします**

### 不足情報

- [ ] よくある問い合わせのトップ10〜20（実績ベース）— FAQの優先順位付けに必要。ないとカバレッジの低いFAQから作り始めるリスクあり
- [ ] マニュアルの最終更新日・バージョン情報 — 古い情報でFAQを作らないために必要
- [ ] ボットが答えてはいけない範囲の明文化 — インシデント・個人情報関連の誤回答防止に必要
- [ ] カテゴリ別の問い合わせ件数内訳 — FAQの優先カテゴリを決定するために必要
- [ ] Ollamaで使用するモデル名・バージョン — チャンクサイズ・プロンプト設計に影響
- [ ] Difyのバージョン（v0.x.x） — 機能の使用可否確認に必要

### 確認事項

- [ ] ボットが答えてはいけない範囲はどこですか？（例: 個人の人事・給与情報、法的判断、インシデント対応の指示など）
- [ ] マニュアルは最新の状態ですか？最終更新日はいつですか？
- [ ] 月200件のうち、カテゴリ別の内訳データはありますか？
- [ ] Ollamaで使用するモデルは決まっていますか？
- [ ] Difyのバージョンを教えていただけますか？
- [ ] 夜間・休日の問い合わせ件数のデータはありますか？（自動化の効果測定の基準になります）
- [ ] セキュリティインシデント（ランサムウェア、情報漏洩等）の問い合わせは、ボットが「即座に内線9999へ電話」と案内することで問題ありませんか？

### 次のステップ

確認事項の回答後、`/kb-design` でナレッジ設計に進みます。

**具体的な作業イメージ（確認事項回答前でも開始可能なこと）:**

1. マニュアルの各章をFAQドラフトに変換（優先: パスワード・VPN・PC・セキュリティ系）
2. 問い合わせログからFAQのQ（質問文）を生成・整理
3. エスカレーションルールの草案作成（インシデント系の定義）

## 付録: カテゴリ別のFAQ化候補

### カテゴリ別の推定FAQ件数と優先度

| カテゴリ | 問い合わせ件数（ログ） | 推定FAQ件数 | 優先度 | 備考 |
|---------|---------------------|------------|--------|------|
| VPN・ネットワーク | 5件 | 6〜8件 | 高 | 在宅勤務増加で需要大 |
| パスワード・アカウント | 5件 | 6〜8件 | 高 | 繰り返し問い合わせの典型 |
| PC・ハードウェア | 6件 | 6〜8件 | 高 | 件数最多。代替機申請等は手順が明確 |
| ソフトウェア・ライセンス | 5件 | 5〜7件 | 中 | 申請フローが主 |
| メール・Teams | 6件 | 6〜8件 | 高 | 容量問題・共有設定は繰り返しやすい |
| セキュリティ | 5件 | 6〜8件 | 高 | インシデント系はエスカレーション必須 |
| プリンター・複合機 | 4件 | 4〜5件 | 中 | 手順が明確。FAQ化しやすい |
| 申請・手続き | 4件 | 5〜7件 | 高 | 新入社員・異動時の繰り返し問い合わせ |
| クラウド・SaaS | 4件 | 4〜6件 | 中 | ポリシー系が多い |
| 障害・インシデント | 4件 | 3〜4件 | 高（特別扱い） | FAQではなくエスカレーションルールとして設計 |

**合計推定: 57〜79件 → 60〜80件と見込む**

### Ollama（ローカルLLM）固有の設計上の注意点

- **チャンクサイズ**: 小さめ（128〜256トークン推奨）。大規模モデルに比べてコンテキスト理解が弱いため、1FAQは短く明確に
- **結論先出し原則の厳守**: ローカルLLMはLost-in-the-Middle問題がより顕著。FAQの結論を必ず冒頭に
