# 最終レポート — 社内ITヘルプデスクチャットボット展開提案

> 作成日: 2026-04-06
> 対象: 社内従業員 約500名
> 環境: Dify Self-hosted + Ollama（ローカルLLM）
> 作成根拠: assess_report.md / design_report.md / build_guide.md / eval_plan.md の総合結果

---

## エグゼクティブサマリー

月約200件のITヘルプデスク問い合わせのうち、繰り返し質問（VPN接続、パスワードリセット、PC操作等）が大半を占めており、3名の担当者が手動対応している現状は非効率かつ夜間・休日対応が不可能である。70件のFAQを搭載したDifyチャットボット（Ollama + Self-hosted環境）を構築することで、月100件（50%）の問い合わせを自動化できると推定される。

**推奨アクション:** 1部門でのPoC（2週間）から開始し、評価後に全社展開へ移行する段階的アプローチを推奨する。初期構築コストは約7人日、月次ランニングコストはOllama利用のためAPI費用ゼロ、サーバー費のみ。

---

## 現状の課題

| 課題 | 内容 | 影響 |
|------|------|------|
| 繰り返し問い合わせが多い | 月200件中、約70%（約140件）が同じ質問の繰り返し | 担当者3名の工数を非効率に消費 |
| 夜間・休日対応不可 | 営業時間外は対応できない | 在宅勤務者・緊急時の対応遅延 |
| FAQ未整備 | マニュアルはあるが従業員向けのFAQ一覧がない | 従業員が自己解決できず問い合わせに頼らざるを得ない |
| 担当者の属人化リスク | 3名体制で知識が担当者に集中 | 欠員時の対応品質低下 |

**現状コストの推計:**
- 月200件 × 平均対応時間15分 = 月50時間の対応工数
- 時給換算3,000円として: 月約15万円の人的コスト
- 3名が繰り返し対応に時間を取られることで本来業務への影響も大きい

## 運用提案

### 誰が使うか

#### 第1フェーズ: IT部門内でのPoC（利用者: IT部門担当者 5〜10名）

- ターゲット: ITヘルプデスク担当者自身がボットを使って動作確認・評価を行う
- 人数: IT部門 5〜10名
- アクセス方法: Dify共有URL（管理画面からPublish → 共有リンク）
- 目的: ボットが正しく動作するか、FAQ内容が適切かを内部検証する

#### 第2フェーズ: パイロット部門展開（利用者: 1部門 約50名）

- ターゲット: 問い合わせが多い部門（例: 営業部、または新入社員の多い部門）
- 人数: 1部門 約50名
- アクセス方法: 社内ポータル（イントラネット）への埋め込みウィジェット or Slack/Teams連携
- 目的: 実利用データ（解決率、フィードバック、未解決質問）を収集して改善する

#### 第3フェーズ: 全社展開（利用者: 全従業員 約500名）

- ターゲット: 全社員
- 人数: 約500名
- アクセス方法: 社内ポータルへのウィジェット埋め込み + Slack/Teamsへのボット統合
- 目的: 月100件（50%）の問い合わせ自動化目標を達成する

### どうアクセスするか

**フェーズ別アクセス方法:**

| フェーズ | アクセス方法 | 工数目安 | メリット |
|---------|------------|--------|--------|
| PoC | Dify共有URL | 0（即日） | 最速。承認不要で即日展開可能 |
| パイロット | 社内ポータルへのiframe埋め込み | 0.5人日 | 利用者が普段使うポータルからアクセス可 |
| 本番 | Slack / Teamsボット統合 | 2〜3人日 | 利用者が最も使うチャットツールで完結 |

**推奨アクセス経路の優先順位:**

1. Slackボット統合（推奨: 最も利用頻度が高い）
   - Slack App設定 → Dify Webhook URLを登録
   - 利用者は普段のSlackから `/ithelp VPNに繋がらない` などと入力するだけ
   
2. 社内ポータルウィジェット埋め込み（推奨: Slack非利用部門向け）
   - Difyの埋め込みコード（iframe）を社内ポータルに挿入
   - PCがブラウザから直接アクセス可能
   
3. Dify共有URL（PoC / 評価フェーズのみ）
   - 管理画面から1クリックで発行可能
   - 認証なしでアクセスできるため、本番運用には不向き（社内VPN内に限定）

| 指標 | 確認内容 | 目標値 |
|------|---------|--------|
| 月間利用件数 | Dify管理画面のログから集計 | 月50件以上（パイロット時） |
| Good/Bad フィードバック比率 | フィードバックボタン（Good/Bad）の比率 | Good率 ≥ 70% |

- `/kb-eval` を再実行してHit Rate@3 / Groundedness / Completenessを計測

- 追加FAQ候補リストを更新

**モニタリング担当者:** IT部門のナレッジ管理担当（同担当者）

**注記:**
- Ollamaのモデルは14BクラスならGPU（VRAM 16GB）推奨。CPU推論でも動作するが応答速度が大幅に低下する（1クエリ30秒〜数分）

- qwen2.5:14b で GPU推論する場合: VRAM 16GB以上（RTX 3080 / RTX 4080 / A100等）
- GPU資産がない場合は elyza:jp8b（8B）モデルでCPU推論も選択肢（応答15〜30秒程度）

## 成功指標と測定方法

### 主要KPI

| KPI | 測定方法 | 目標値 | 測定頻度 |
|-----|---------|--------|---------|
| ボット解決率 | ヘルプデスク問い合わせ件数の削減率 | 月100件（50%削減） | 月次 |
| Good評価率 | Difyフィードバックボタン（Good/Bad比率） | ≥ 70% | 月次 |
| エスカレーション率 | フォールバック（「内線9999へ」）の割合 | ≤ 20% | 月次 |
| 夜間・休日解決率 | 夜間・休日に問い合わせがあった場合の自動解決率 | ≥ 50% | 月次 |

### 測定方法の詳細

#### 1. ボット解決率の測定

```
測定方法:
  (A) ベースライン計測: パイロット前1ヶ月のヘルプデスク問い合わせ件数を記録
  (B) 運用後計測: ボット利用件数（Difyログから）+ ヘルプデスク実問い合わせ件数を毎月記録
  (C) 計算: ボット解決率 = ボット利用件数 / (ボット利用件数 + ヘルプデスク実問い合わせ件数)

目標: ≥ 50%（月100件以上がボットで完結）
```

#### 2. Good評価率の測定

```
測定方法:
  Dify管理画面 → アプリのログ → フィードバック一覧
  Good率 = Good件数 / (Good件数 + Bad件数)

Bad評価の活用:
  Bad評価がついた質問を月次で確認
  → 同じ質問が3件以上Bad評価 → FAQの修正候補リストに追加
```

#### 3. 四半期評価（kb-eval再実行）

```
実施タイミング: 全社展開後3ヶ月ごと（1月・4月・7月・10月）
評価内容:
  - テスト質問20問（eval_plan.md のセットを継続使用）
  - 新たに追加されたFAQのRetrieval評価
  - ギャップ分析（直近の問い合わせログから新しいトピックを確認）

判断ポイント:
  Hit Rate@3 が 80%を下回ったらFAQ見直し
  Groundedness Score が 90%を下回ったらプロンプト調整
  解決率が 50%を下回ったらFAQ追加を優先
```

## リスクと対策

| リスク | 発生確率 | 影響度 | 対策 |
|--------|---------|--------|------|
| Ollamaモデルの日本語品質が不十分 | 中 | 高 | 事前にqwen2.5:14bで日本語回答品質を確認。品質不足ならelyza:jp8bを試す |
| 緊急インシデント（ランサムウェア等）への誤回答 | 低 | 最高 | システムプロンプトに緊急キーワードを明示的に列挙。PoC段階で必ずテスト |
| FAQの陳腐化（古い情報での回答） | 高 | 中 | 月次レビューのルール化。制度変更時の即時更新フローを事前に決めておく |
| ユーザーの利用低下（定着しない） | 中 | 高 | Slack統合で普段使うツールからアクセス可能にする。Good/Bad評価でフィードバックを収集し改善サイクルを回す |
| 検索精度の低下（FAQ追加・削除時） | 中 | 中 | FAQを大きく変更した後は必ずRetrieval Testで確認する |
| サーバー障害（Single Point of Failure） | 低 | 中 | Dify + Ollamaサーバーのバックアップ設定。障害時はヘルプデスクへのフォールバック案内を維持 |
| ハルシネーション発生 | 中 | 高 | Temperature低め設定（0.2）+ システムプロンプトで制約強化。Groundedness評価を四半期ごとに実施 |

## 提案A（推奨）: 段階的展開（PoC → パイロット → 全社）

**推奨理由:** 最もリスクが低く、各フェーズで品質確認してから次に進めるため、失敗のコストを最小化できる。
