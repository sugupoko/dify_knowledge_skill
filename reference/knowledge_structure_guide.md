# 3層ナレッジ設計ガイド — FAQ + 支援文書 + 運用情報

## なぜFAQだけでは足りないか

FAQだけでボットを構築すると、以下の問題が起きる:

| 症状 | 原因 |
|------|------|
| 「手順を教えてください」に答えられない | FAQは結論のみで手順の詳細がない |
| 「これは社内ルール的にOKですか？」に答えられない | 規程・ポリシーがナレッジにない |
| 回答が正しいか検証できない | 参照元がない |
| 古い情報で回答する | 更新管理の仕組みがない |
| 答えてはいけない質問に答えてしまう | 制約・エスカレーションルールがない |

**FAQは「直接回答」を担う層。それ以外の層がないと、ボットは"それっぽいが使えない"回答を返す。**

---

## 3層の構造

```
┌─────────────────────────────────────────────┐
│  層1: FAQ（直接回答層）                       │
│  ・ユーザーの質問に直接答える短い知識ユニット    │
│  ・1FAQ1テーマ、結論先出し                    │
│  ・検索の第一候補                             │
├─────────────────────────────────────────────┤
│  層2: 支援文書（詳細・手順層）                 │
│  ・手順書・マニュアル                         │
│  ・規程・ポリシー                             │
│  ・入力検証ルール                             │
│  ・用語集・別名辞書                           │
│  ・FAQだけでは答えきれない時の詳細情報          │
├─────────────────────────────────────────────┤
│  層3: 運用情報（メタ層）                       │
│  ・エスカレーションルール                      │
│  ・更新責任者・最終更新日                      │
│  ・有効期間                                  │
│  ・ボットが答えてはいけない範囲                 │
└─────────────────────────────────────────────┘
```

## 層1: FAQ（直接回答層）

### 目的
ユーザーの質問に対して、最も短い経路で正確な回答を提供する。

### 設計原則
- 1FAQ = 1テーマ = 1チャンク
- 結論を冒頭に（1-2文で回答）
- 条件・例外を明示
- 次の行動を示す
- 参照元を記録

### 粒度の目安
- 128〜512 tokens（Q&A合わせて）
- 多すぎず少なすぎない: 1つの質問に自己完結して答えられる分量

### 手段別の扱い

| 手段 | FAQの扱い |
|------|----------|
| チャットボット | 単一ナレッジベースにFAQを格納 |
| チャットフロー | テーマ別に複数ナレッジベースに分割 |
| エージェント | FAQ + メタデータ（カテゴリ、対象業務、信頼度） |
| 検索型 | FAQ + セマンティック検索（retriever_resource 表示ON） |
| ワークフロー | FAQではなくテンプレート・データ取得ロジックで構成 |

### 詳細
→ `faq_design_guide.md` を参照

## 層2: 支援文書（詳細・手順層）

### 目的
FAQだけでは答えきれない詳細情報を提供する。「もっと詳しく」「手順を教えて」に対応する。

### 含めるべき文書の種類

#### 手順書・マニュアル
```
用途: 「どうやるの？」に答える
設計:
  - チャットボット: 全体を1チャンクで持つか、FAQから参照リンク
  - チャットフロー: テーマ別に分割してステップ処理
  - エージェント: ステップ単位で分割（前提条件・次のステップへの参照付き）

チャンキング例:
  ステップ1: 申請フォームにアクセスする（URL: xxx）
    前提: 社内ネットワークに接続していること
    次: ステップ2へ
  ステップ2: 必要事項を入力する
    必須項目: 氏名、部署、金額、日付
    次: ステップ3へ
```

**ニーズ別の持たせ方（手順書）:**

| ユーザーのニーズ | 最適な手段 | ナレッジの持たせ方 |
|----------------|----------|-----------------|
| 手順の全体像を把握したい（「経費精算の流れは？」） | 検索型 | 手順書全体を1チャンクまたはセクション単位で保持。検索でそのまま表示する |
| 今の作業の次のステップを知りたい（「申請した、次は？」） | チャットフロー or チャットボット | ステップ単位でチャンク化。前のステップ・次のステップへの参照を含める |
| 手順の途中で条件が分からない（「この場合は承認いる？」） | チャットフロー | 条件ごとに分岐するフロー設計。条件分岐をナレッジではなくフロー制御で担う |

→ 参考: CHI 2022 ([How to Guide Task-oriented Chatbot Users](https://dl.acm.org/doi/10.1145/3491102.3501941)), Conversation Routines ([arXiv 2025](https://arxiv.org/html/2501.11613v3))

#### 規程・ポリシー
```
用途: 「ルール的にOK？」に答える
設計:
  - 条文単位でチャンキング
  - 例外規則は別チャンクにして関連性を持たせる
  - 「いつから有効か」「誰に適用されるか」のメタデータ

チャンキング例:
  条文: 出張旅費は実費精算とする。ただし、交通費の上限は以下の通り。
  適用対象: 全社員
  例外: 役員は別途規程による
  有効期間: 2026-04-01〜
  参照: 就業規則 第15条
```

#### 入力検証ルール
```
用途: 「この値は正しい？」「何を入力すればいい？」に答える
設計:
  - フィールドごとに1チャンク
  - 許容値、必須/任意、形式を明示

例:
  フィールド: 申請金額
  型: 数値
  範囲: 1円〜100万円
  必須: はい
  注意: 税込み金額で入力。100万円超は部長承認が必要
```

#### 用語集・別名辞書
```
用途: 検索精度の向上 + 「XXとは何？」に答える
設計:
  - 1用語 = 1チャンク
  - 正式名称、略語、俗称、英語名を含める

例:
  正式名称: 年次有給休暇
  略語: 有休、有給
  俗称: 年休
  英語: Annual Paid Leave
  定義: 労働基準法第39条に基づく...
```

## 層3: 運用情報（メタ層）

### 目的
ボットの動作を制御し、適切な運用を維持するための情報。

### 含めるべき情報

#### エスカレーションルール
```
用途: 「人に回すべき時」をボットが判断する材料
設計:
  - 条件: いつ人に回すか
  - 引き継ぎ先: 誰に回すか（部署、担当者、連絡先）
  - 引き継ぎ情報: 何を添えるか（質問内容、試した対処、ユーザー情報）

例:
  条件: 以下のいずれかに該当する場合、人間のオペレーターに引き継ぐ
    - 個人情報に関する質問
    - クレームや苦情
    - FAQに該当する情報がない場合が3回連続
    - ユーザーが「人と話したい」と言った場合
  引き継ぎ先: カスタマーサポート（内線1234）
  引き継ぎ情報: 会話履歴の要約、ユーザーの質問、試した回答
```

#### 更新責任者・最終更新日
```
用途: ナレッジの鮮度管理
設計:
  - 各FAQ/文書に更新責任者と最終更新日を付与
  - 更新期限を設定（例: 6ヶ月ごとにレビュー）
  - 古い情報のフラグ付け

例:
  FAQ ID: FAQ-001
  更新責任者: 総務部 田中
  最終更新日: 2026-03-15
  次回レビュー: 2026-09-15
  ステータス: 有効
```

#### ボットが答えてはいけない範囲
```
用途: ハルシネーション防止 + コンプライアンス
設計:
  - 明示的な禁止事項リスト
  - 回答を控えるべきトピック
  - 断定してはいけない条件

例:
  禁止事項:
    - 個別の人事評価に関する回答
    - 給与の具体的な金額の回答
    - 法的アドバイスに相当する回答
    - 医療上のアドバイス
  回答を控えるべき場合:
    - ナレッジベースに該当情報がない場合 → 「該当する情報が見つかりません」
    - 情報が古い可能性がある場合 → 「最新情報は担当部署にご確認ください」
```

## 手段別の必要度

| 層 | 内容 | チャットボット | チャットフロー | エージェント | 検索型 | ワークフロー |
|----|------|:----------:|:----------:|:--------:|:----:|:--------:|
| **層1** | FAQ | 必須 | 必須 | 必須 | 必須 | 不要 |
| **層2** | 手順書 | リンク程度 | ステップ処理 | ステップ分割必須 | 参考文書 | テンプレートとして必須 |
| **層2** | 規程・ポリシー | リンク程度 | 条件分岐で使用 | 判断根拠として必須 | 参考文書 | 不要 |
| **層2** | 用語集 | 検索精度向上 | 分類精度向上 | ツール選択に有効 | 検索精度向上 | 不要 |
| **層3** | エスカレーション | FAQ内に含める | フロー分岐条件 | 自律判断の制約 | 不要 | 不要 |
| **層3** | 更新管理 | あると良い | あると良い | 必須 | あると良い | 必須 |
| **層3** | 禁止事項 | プロンプトで制御 | フロー設計で制御 | ナレッジとして必須 | 不要 | 不要 |

## 実務的な進め方

### Step 1: 層1（FAQ）を先に作る
- `faq_design_guide.md` に従ってFAQを設計
- まずこれだけでボットを動かして評価する
- 評価結果から層2・層3の必要性を判断

### Step 2: 不足を特定する
- 「手順を教えて」系の質問に答えられない → 層2（手順書）を追加
- 「ルール的にOK？」に答えられない → 層2（規程）を追加
- 答えてはいけない質問に答えている → 層3（禁止事項）を追加
- 古い情報で回答している → 層3（更新管理）を追加

### Step 3: 段階的に層を追加する
- 一度に全部作らない
- 評価結果を見ながら、不足している層を優先的に追加
- 各層の追加後に必ず再評価
