# ナレッジ設計仕様書テンプレート（spec.md）

各バージョンフォルダ内に置く。**最新バージョンの spec.md が「今の正」。**
過去バージョンの spec.md と比較すれば、何が変わったかがわかる。

**全ての設計判断・仮定に情報源の番号（Ref）を付けること。** 後から「この設計はどこから来たのか」を追跡できるようにする。

---

以下を `workspace/<project>/v1/spec.md` にコピーして使う。
バージョンが上がったら `v2/spec.md` に複製して更新する。

```markdown
# ナレッジ設計仕様書

バージョン: v1
最終更新: YYYY-MM-DD
プロジェクト: [プロジェクト名]

## 情報源

全ての設計判断・仮定の根拠を追跡するための一覧。各項目の Ref 列がここに紐づく。

| Ref | 種類 | 文書・情報源 | 取得日 | 備考 |
|-----|------|------------|--------|------|
| R1 | データ | faq_existing.xlsx | YYYY-MM-DD | クライアントから受領 |
| R2 | ヒアリング | 担当者ヒアリング（YYYY-MM-DD） | YYYY-MM-DD | 要件・制約の確認 |
| R3 | 文書 | 社内規程集.pdf | YYYY-MM-DD | 業務ルールの根拠 |
| R4 | 仮定 | 分析者の仮定 | YYYY-MM-DD | 実データなし |

## 目標

- 主目標: [例: 社内ヘルプデスクの問い合わせを70%自動化]  ← Ref: R2
- 副目標: [例: 回答品質の均一化、属人化の解消]  ← Ref: R2
- 優先順位: [主目標 > 副目標]  ← Ref: R2
- 対象ユーザー: [例: 社員全員（約500名）]  ← Ref: R2
- 月間問い合わせ件数: [例: 100件/月]  ← Ref: R1

## 推奨手段と根拠

- 推奨手段: [例: チャットフロー]  ← Ref: R2
- 根拠: [例: FAQ件数が150件以上、部門別ルーティングが必要]  ← Ref: R1, R2
- 代替手段: [例: チャットボット（1部門限定PoCの場合）]

## ナレッジ設計テーブル

| # | 種類 | 設定値 | 根拠 | Ref |
|---|------|--------|------|-----|
| K1 | ナレッジベース構成 | [例: 総務/IT/経理の3分割] | 部門別ルーティングが必要 | R2 |
| K2 | チャンキング単位 | [例: 1FAQ = 1チャンク] | FAQ形式のため | R4 |
| K3 | Chunk Size | [例: 500トークン] | 平均FAQ長さに基づく | R1 |
| K4 | Overlap | [例: 0] | FAQ間の独立性が高い | R4 |
| K5 | Retrieval Mode | [例: Semantic Search] | 言い換えに対応するため | R4 |
| K6 | Top K | [例: 3] | PoCフェーズの標準設定 | R4 |
| K7 | Score Threshold | [例: 0.5] | PoCフェーズの標準設定 | R4 |
| K8 | 推定FAQ件数 | [例: 150件] | 既存85件+マニュアルから65件推定 | R1 |

## プロンプト設計テーブル

| # | セクション | 内容 | 根拠 | Ref |
|---|-----------|------|------|-----|
| P1 | 役割定義 | [例: 「あなたは社内ヘルプデスクのアシスタントです」] | 回答スコープを明確化 | R2 |
| P2 | 回答ソース制限 | [例: 「ナレッジベースの情報に基づいて回答してください」] | ハルシネーション防止 | R4 |
| P3 | フォールバック | [例: 「該当情報がない場合は担当部署にお問い合わせくださいと伝える」] | エスカレーション設計 | R2 |
| P4 | 回答形式 | [例: 「3〜5文で簡潔に、結論を先に述べる」] | 利用者のUX要件 | R2 |
| P5 | 禁止事項 | [例: 「個人の給与・評価情報は回答しない」] | プライバシー・ポリシー | R3 |
| P6 | エスカレーション条件 | [例: 「クレーム・法的問題・医療相談は人に回す」] | リスク管理 | R2 |

## 仮定テーブル

| # | 仮定 | 値 | 根拠 | Ref | 検証状況 |
|---|------|-----|------|-----|---------|
| A1 | [例: 月間問い合わせの内訳（部門別）] | [例: 総務40%、IT40%、経理20%] | 担当者の感覚値 | R4 | 未検証 |
| A2 | [例: 既存FAQのボット転用可能率] | [例: 30%（残りは再設計が必要）] | サンプル確認 | R1 | 部分検証済 |
| A3 | [例: 1FAQ生成にかかる工数] | [例: 3分/件] | 類似案件の実績 | R4 | 未検証 |

## 受け入れ要件

### 利用者の利用方法
- アクセス方法: [例: 社内ポータルへのウィジェット埋め込み / Slack連携]  ← Ref: R2
- 利用端末: [例: PC / スマートフォン / 印刷]  ← Ref: R2
- 利用頻度: [例: 1日複数回 / 週1-2回]  ← Ref: R2

### 表示形式
- 回答の長さ: [例: 3〜5文、簡潔に]  ← Ref: R2
- 引用元の表示: [例: あり（「社内規程集 第○条」） / なし]  ← Ref: R2
- 改行・箇条書きの使用: [例: 手順は番号付きリスト、通常回答は文章]  ← Ref: R2
- エスカレーション時の案内文: [例: 「詳細は総務部（内線XXX）にお問い合わせください」]  ← Ref: R2

### FAQ更新方法
- 更新担当者: [例: 各部門の担当者 / DX推進室]  ← Ref: R2
- 更新頻度: [例: 月次定期レビュー + 随時追加]  ← Ref: R2
- 更新トリガー: [例: 規程改定、新制度導入、未回答率が10%を超えた時]  ← Ref: R2
- 更新手順: [例: CSV編集 → Difyアップロード → 動作確認]  ← Ref: R4

（v1 では省略。v2 以降で記載する）
```
