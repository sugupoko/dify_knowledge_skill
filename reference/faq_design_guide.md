# チャットボット向けFAQ設計ガイド

## 参考文献

| 文献 | 何の参考にしたか |
|------|----------------|

| Liu et al. Lost in the Middle (TACL 2024) | 長い文書の中ほどの情報が使われにくい。FAQを長い説明文のまま持つのは弱い |
| Es et al. RAGAS (EACL 2024) | RAGの評価。「自然に見えるか」だけでなく retrieval / groundedness を分けて見る |
| Pinecone RAG ガイド | hybrid search、rerank |

| Dify Docs | Knowledge Base / Retrieval Test / Citation / Chunk Settings |

| [CHI 2022: How to Guide Task-oriented Chatbot Users, and When](https://dl.acm.org/doi/10.1145/3491102.3501941) | タスク指向チャットボットのガイダンス。ステップバイステップ案内 + タスク開始時の提示がタスク遂行率を上げる |

## FAQ と手順書の違い

FAQと手順書は目的が異なる。混同すると設計が崩れる。

| | FAQ | 手順書 |
|---|-----|-------|
| 構造 | 1問1答（独立した知識ユニット） | ステップのシーケンス（前後の依存がある） |
| ユーザーのニーズ | 「答えを知りたい」 | 「次に何をすればいいか知りたい」 |
| 最適な提供手段 | チャットボット、検索型 | チャットフロー（ステップ案内）、検索型（全体把握） |
| チャンキング | 1FAQ = 1チャンク | ステップ単位（前後ステップへの参照付き） |
| 条件分岐 | FAQ内で例外を記載 | フロー制御で分岐する（ナレッジに頼らない） |

**手順が5ステップ以上になるものは、FAQとして無理に収めず手順書（層2）として別管理する。**  
→ 参考: Conversation Routines ([arXiv 2025](https://arxiv.org/html/2501.11613v3)) — 手順的なタスクは「1ステップずつ提示し、ユーザーの確認を待つ」設計が有効

## 核心: 人向けFAQ ≠ ボット向けFAQ

| | 人向けFAQ | ボット向けFAQ |
|---|----------|-------------|
| 前提 | 人が読んで理解する | 検索されて切り出されて使われる |
| 長さ | 多少長くてもOK | 短い知識ユニットが有利 |
| 構造 | 背景から説明しても読める | 結論が先でないと使いにくい |
| 曖昧さ | 人が文脈を補える | そのまま誤答につながる |
| 複数論点 | 1項目に混ぜても読める | 検索ノイズになる |

**人向けFAQは「読めばわかる」でよい。ボット向けFAQは「拾われても崩れない」が必要。**

## FAQ1項目に持たせるべき情報

| 要素 | 説明 | 表示 | 内部データ |
|------|------|------|-----------|
| 質問文 | ユーザーが実際に言いそうな表現 | 表示 | 必須 |
| 短い結論 | 最初の1-2文で答える | 表示 | 必須 |
| 条件 | いつ当てはまるか | 表示 | 必須 |
| 例外 | 当てはまらないケース | 表示 | 必須 |
| 次の行動 | ユーザーが次にやること | 必要なら表示 | 必須 |
| エスカレーション条件 | 人に回す条件と引き継ぎ先 | 必要なら表示 | 必須 |
| 参照元 | 文書名・章・節・ページ | リンクで | 必須 |
| 言い換え・別名 | 略語、俗称、誤記候補 | 非表示 | 必須 |
| ボット向け注意点 | 断定禁止条件など | 非表示 | 推奨 |

**表示用FAQは短く、ボット用の知識データは厚く持つ。**

## 1FAQ 1テーマの原則

### なぜ1テーマ1ユニットか

RAGではユーザーの質問に対してベクトル類似度で検索する。1項目に複数テーマが含まれると:
- 質問Aで検索した時に、テーマBの部分がノイズとして回答に混入する
- 質問Bで検索した時に、テーマAの部分がスコアを押し上げて誤ヒットする
- LLMが複数テーマの中から適切な部分だけ抽出する負荷が増える

### 分割の目安

| 状況 | 判断 |
|------|------|
| 1つの質問に1つの回答で完結 | 分割不要 |
| 「AとBについて」で始まる | A, B それぞれ独立のFAQに分割 |
| 条件で回答が大きく変わる | 条件ごとにFAQを分割（「正社員の場合」「契約社員の場合」） |
| 手順が5ステップ以上 | FAQとしては概要のみ、詳細は支援文書（層2）で持つ |

## 結論先出しの原則

### なぜ結論を先に出すか

Liu et al. (2024) "Lost in the Middle" の知見:
- LLMは長い入力の先頭と末尾の情報を優先的に使う
- 中間部分の情報は活用されにくい

FAQ項目が「背景→説明→結論」の構造だと、結論がLLMに拾われにくい。

### 構造テンプレート

```
■ 結論（1-2文で回答）
■ 条件（いつ当てはまるか）
■ 例外（当てはまらないケース）
■ 次の行動（ユーザーがやること）
■ 参照元（文書名・ページ）
```

## 良いFAQと悪いFAQの例

### 例1: ログインできない

**悪い例:**
> Q. アカウントにログインできません。
> ログインできない場合は、入力内容やご利用環境をご確認ください。必要に応じて登録情報の確認や再設定を行ってください。詳細についてはヘルプページもあわせてご確認ください。

問題: 結論が弱い / 条件分岐がない / 何をすればよいか分からない

**良い例:**
> Q. パスワードを忘れてログインできないときはどうすればいいですか？
> ログイン画面の「パスワードを忘れた方」から再設定してください。
> 登録メールアドレスが使えない場合は、サポート窓口（内線1234）に連絡してください。

内部データ:
- 条件: パスワード忘れによるログイン不可
- 例外: 登録メールアドレスが利用できない場合はセルフサービス不可
- 次の行動: 再設定ページを案内 / サポート窓口を表示
- 参照元: アカウント設定ヘルプ 2章

### 例2: 配送

**悪い例:**
> Q. 配送について教えてください。
> 配送は通常、各地域ごとの配送スケジュールに従って行われます。天候や交通状況などにより到着までの日数が変動することがあります。

問題: 抽象的 / 結論がない / 判断軸がない

**良い例:**
> Q. 注文した商品はいつ届きますか？
> 通常は発送から2〜4日で届きます。
> 離島・一部地域ではそれ以上かかる場合があります。
> 発送完了メールに追跡番号がある場合は、配送状況を確認してください。

### 例3: FAQに向かないもの

**悪い例:**
> Q. 特殊事情がある場合はどうしたらよいですか？
> ケースにより異なりますので、個別にご相談ください。

問題: FAQではなく相談窓口案内。類似質問がこの項目に吸われてしまう。

**良い例:**
> Q. 通常ルールに当てはまらない場合はどうすればいいですか？
> 以下のいずれかに当てはまる場合は、サポート窓口（内線1234）に連絡してください。
> - 条件Aが基準外
> - 条件Bが超過
> - 必要項目が不足
> 連絡時に必要な情報: 対象ID、日時、条件の詳細

## 悪いFAQの典型パターン

| パターン | 特徴 |
|---------|------|
| **ふわっとしている** | 「場合があります」「必要に応じて」「詳細は別紙」が多く、結論や判断軸がない |
| **論点が多すぎる** | 1項目に受付時間・例外・提出先・連絡先が全部入っている |
| **背景説明から入る** | 人向け文書では許されても、ボットでは検索・切り出し時に不利 |
| **FAQで答えられないものをFAQ化** | 個別判断が必要なものを無理にFAQにしている |
| **参照先がない** | 根拠が追えない。改善も監査もできない |

## 良いFAQのチェックリスト

### 必須

- [ ] 1FAQ1テーマになっている
- [ ] 質問文がユーザーの言い方に近い
- [ ] 回答の最初の1-2文で結論が分かる
- [ ] 条件と例外が分かれている
- [ ] FAQだけで完結しない場合、次の行動が書いてある
- [ ] 人に回すべき条件があるなら明示している
- [ ] 根拠となる文書名や参照先を持っている

### あると強い

- [ ] 略語、別名、言い換えを持っている
- [ ] FAQ本文と内部用の補足データを分けている
- [ ] FAQで断定しすぎないための注意点がある
- [ ] FAQ化に向かないものを無理にFAQにしていない
- [ ] 手順書、ルール、用語集などの支援文書が別にある

### 危ないサイン

- 1項目に複数論点が入っている
- 結論が最後まで出てこない
- 「場合があります」が多く判断軸がない
- 「詳細は別紙」とだけ書いてある
- FAQというより相談窓口案内になっている
- 根拠がどこか分からない
