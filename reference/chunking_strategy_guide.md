# チャンキング戦略ガイド — 文書タイプ別・手段別

## チャンキングの基本原則

### なぜチャンキングが重要か

RAGでは、検索結果のチャンクがそのままLLMのコンテキストに入る。チャンクの粒度が:
- **大きすぎる**: 関係ない情報がノイズとして混入。Lost in the Middle問題が顕著に。
- **小さすぎる**: 文脈が失われ、チャンクだけでは意味が通じない。
- **適切**: 1つのテーマについて自己完結した情報が含まれる。

### 基本方針

```
意味のまとまり（セマンティックチャンキング）で分割する。
文字数ではなく「1チャンク = 1つの問いに答えられる単位」を目指す。
```

## 文書タイプ別のチャンキング戦略

### 1. FAQ文書

| 項目 | 推奨 |
|------|------|
| 分割単位 | 1FAQ = 1チャンク |
| トークン目安 | 128〜512 tokens |
| 形式 | Q: ... A: ... の形を保つ |
| メタデータ | カテゴリ、言い換え、参照元 |

**ポイント:**
- Q&A形式はembeddingの質が高い（質問文がユーザーの聞き方に近い）
- 複数論点が1FAQに含まれている場合は先に分割してからチャンク化
- 言い換え・別名はメタデータまたはQ文に含める

```
良いチャンク:
  Q: 経費精算の提出期限はいつですか？
  A: 毎月末日が締め切りです。月末が休日の場合は前営業日になります。
  条件: 全社員共通
  例外: 海外出張の場合は帰国後10営業日以内
  参照: 経費精算規程 第5条

悪いチャンク:
  経費精算について
  提出期限は毎月末日です。上限額は正社員10万円、契約社員5万円です。
  承認フローは直属上長→部長→経理です。領収書は原本を添付してください...
  → 論点が混在しており、どの質問にもノイズになる
```

### 2. マニュアル・手順書

| 項目 | チャットボット | 検索型 | チャットフロー | エージェント | ワークフロー |
|------|:----------:|:----:|:----------:|:--------:|:--------:|
| 分割単位 | 全体1チャンク or セクション | セクション単位 | セクション単位 | ステップ単位 | テンプレート単位 |
| トークン目安 | 256〜1024 | 256〜512 | 256〜512 | 128〜512 | 制限なし |
| 形式 | 見出し + 本文 | 見出し + 本文 | 見出し + 本文 | ステップ + 前提 + 次ステップ | 変数プレースホルダー形式 |

**チャットボットの場合:**
```
チャンク例:
  ## 経費精算の申請手順
  1. 社内ポータルにログイン → 経費精算メニューを選択
  2. 申請フォームに必要事項を入力
  3. 領収書をスキャンして添付
  4. 直属上長に承認依頼を送信
  ※ 10万円以上の場合は部長承認も必要
```

**エージェントの場合:**
```
チャンク1:
  ステップ1: 社内ポータルにログインする
  URL: https://portal.example.com/expense
  前提: 社内ネットワークに接続していること
  次のステップ: ステップ2（申請フォーム入力）

チャンク2:
  ステップ2: 申請フォームに必要事項を入力する
  必須項目: 氏名、部署、金額（税込）、日付、用途
  前のステップ: ステップ1（ログイン）
  次のステップ: ステップ3（領収書添付）
```

### 3. 規程・ポリシー

| 項目 | 推奨 |
|------|------|
| 分割単位 | 条文単位 or セクション単位 |
| トークン目安 | 128〜512 tokens |
| 形式 | 条文番号 + 本文 + 適用対象 |
| メタデータ | 有効期間、適用対象、例外条文への参照 |

```
良いチャンク:
  条文: 就業規則 第15条（出張旅費）
  本文: 出張旅費は実費精算とする。ただし、交通費の上限は以下の通り。
    - 国内: グリーン車以下の実費
    - 海外: エコノミークラス実費
  適用対象: 全社員（役員を除く）
  例外参照: 第15条の2（役員の出張旅費）
  有効期間: 2026-04-01〜

悪いチャンク:
  就業規則より抜粋
  出張旅費は実費精算です。交通費は...宿泊費は...日当は...食事手当は...
  → 複数の論点が1チャンクに混在
```

### 4. Q&Aログ・問い合わせ履歴

| 項目 | 推奨 |
|------|------|
| 分割単位 | 1質問1チャンク（会話全体ではなく） |
| トークン目安 | 128〜512 tokens |
| 形式 | 質問 + 回答（要約） |
| 前処理 | 個人情報の除去、類似QAの統合 |

```
前処理のステップ:
  1. 個人情報を除去（氏名、メールアドレス等）
  2. 質問を正規化（言い換えの統一）
  3. 類似QAを統合（同じ質問に対する複数の回答を最適な1つに）
  4. FAQフォーマットに変換
```

### 5. スプレッドシート・テーブルデータ

| 項目 | 推奨 |
|------|------|
| 分割単位 | 行単位 or 小テーブル単位 |
| トークン目安 | 64〜256 tokens |
| 形式 | キー: 値 の形式 or Markdown表 |

```
良いチャンク（行単位）:
  商品名: ワイヤレスイヤホン Pro X
  価格: 12,800円（税込）
  納期: 2-4営業日
  カラー: ブラック / ホワイト / ネイビー
  注意: 防水性能はIPX4のため、水没には対応していません

悪いチャンク:
  [テーブル全体をそのまま1チャンクに]
  → 数百行のテーブルでは検索精度が著しく低下
```

## Difyでのチャンキング設定

### Difyの設定項目

| 設定 | 説明 | 推奨値 |
|------|------|--------|
| **Chunk Size** | チャンクの最大サイズ | 500-1000 tokens |
| **Chunk Overlap** | チャンク間のオーバーラップ | 50-100 tokens |
| **Indexing Mode** | インデクシング方式 | High Quality（推奨） |
| **Embedding Model** | 埋め込みモデル | text-embedding-3-small or text-embedding-ada-002 |

### 推奨設定（文書タイプ別）

| 文書タイプ | Chunk Size | Overlap | 備考 |
|-----------|-----------|---------|------|
| FAQ (Q&A形式) | 500 | 0 | Q&Aペアで区切る。オーバーラップ不要 |
| マニュアル | 800 | 100 | セクション区切りを尊重 |
| 規程 | 500 | 50 | 条文単位で区切る |
| Q&Aログ | 500 | 0 | 1QAペアで区切る |

### 手順書チャンキングの注意点

手順書は「ステップの途中で切れると不完全・誤解を招く」という点で他の文書より繊細。

- **チャンクがステップをまたぐのを避ける**: ステップ3の前半と後半が別チャンクになると、LLMが不完全な手順を正しいものとして回答するリスクがある（Advanced Chunking for RAG, [PMC 2025](https://pmc.ncbi.nlm.nih.gov/articles/PMC12649634/)）
- **意味的一貫性を保つ**: 固定長チャンキングより、セクション・ステップ境界を優先した分割が精度向上に寄与する（Reconstructing Context, [arXiv 2025](https://arxiv.org/abs/2504.19754)）
- **ステップ状態の管理**: ステップ間の依存関係（前提・次ステップ参照）をメタデータで保持すると、複数ステップにまたがる案内が改善される（HierTOD, [VLDB 2025](https://www.vldb.org/2025/Workshops/VLDB-Workshops-2025/DaSH/DaSH25_6.pdf)）

### Difyの自動チャンキングの限界

Difyの自動チャンキングは文字数ベースで分割するため:
- 意味の途中で切れることがある
- FAQ形式の場合、Q と A が別チャンクに分かれることがある

**対策:**
1. FAQ形式のデータは事前にQ&Aペアごとに整形してからアップロード
2. マニュアルはセクション単位に事前分割
3. 「カスタム区切り文字」設定で `---` や `##` を区切りとして指定

## チャンキング品質のチェックリスト

- [ ] 各チャンクは1つのテーマについて自己完結しているか
- [ ] チャンクだけを読んで意味が通じるか（文脈の喪失がないか）
- [ ] Q&Aペアが分割されていないか
- [ ] 条件・例外が本文と同じチャンクに含まれているか
- [ ] メタデータ（カテゴリ、参照元）が付与されているか
- [ ] 個人情報や機密情報が適切に除去されているか
- [ ] チャンクサイズが極端に大きい/小さいものがないか

## トラブルシューティング

### 検索精度が低い
```
原因と対策:
  ├── チャンクが大きすぎる → サイズを縮小、テーマ別に分割
  ├── チャンクが小さすぎる → 文脈を含めるよう拡大
  ├── Q&Aが分割されている → 事前にQ&Aペアで整形
  ├── 言い換えが不足 → メタデータに言い換え・別名を追加
  └── Embedding品質が低い → High Qualityモードに変更
```

### 回答が冗長
```
原因と対策:
  ├── チャンクに複数論点 → 1テーマ1チャンクに分割
  ├── チャンクが長すぎる → サイズを縮小
  └── top_kが多すぎる → top_kを3-5に設定
```
