## 手順的な質問の評価（手順書・マニュアル）

FAQ（1問1答）の評価とは別に、手順案内には固有の評価観点が必要。

| 観点 | 何を見るか | 目標 |
|------|---------|------|
| **ステップの完全性** | 必要なステップが全て含まれているか | 省略・脱落ゼロ |
| **順序の正確性** | ステップが正しい順番で提示されているか | 誤順序ゼロ |
| **条件分岐の正確性** | 条件に応じて正しいステップに案内しているか | 分岐ミスゼロ |
| **前提条件の提示** | 各ステップの前提（環境・権限など）が伝わるか | 省略率 10%以下 |

**テスト方法:**
```
テストケース例:
  - 「経費精算の申請手順を教えて」→ 全ステップが順番通りに含まれるか
  - 「申請フォームに入力した。次は？」→ 正しい次ステップのみが返るか
  - 「10万円超の申請の場合、承認フローは？」→ 条件に対応した分岐が正しいか
```

→ 参考: CHI 2022 ([How to Guide Task-oriented Chatbot Users](https://dl.acm.org/doi/10.1145/3491102.3501941)), HierTOD ([VLDB 2025](https://www.vldb.org/2025/Workshops/VLDB-Workshops-2025/DaSH/DaSH25_6.pdf))

## 評価の4軸

### 概要

「回答が自然に見えるか」だけでは不十分。以下の4つを分けて評価する。

| 軸 | 何を見るか | なぜ分けるか |
|----|---------|------------|
| **Retrieval（検索精度）** | 正しいFAQを拾えたか | 検索が外れていれば、後段のLLMがいくら優秀でも正しい回答は出ない |
| **Groundedness（根拠性）** | 拾ったFAQに基づいて答えているか | 正しいFAQを拾っても、LLMがハルシネーションする可能性がある |
| **Completeness（網羅性）** | 必要な条件・例外を落としていないか | 結論は正しいが条件の抜け落ちでミスリードになる |
| **Resolution（解決率）** | ユーザーが自己解決できたか | 最終的なビジネス指標。上記3つが改善手段、これが目標 |

---

## 1. Retrieval（検索精度）の評価

### 何を測るか
テスト質問を投げた時に、top_kの検索結果に正解のFAQが含まれるか。

### 指標

| 指標 | 説明 | 計算方法 |
|------|------|---------|
| **Hit Rate@K** | top_Kに正解が1つでも含まれる割合 | 正解含むクエリ数 / 全��エリ数 |
| **MRR（Mean Reciprocal Rank）** | 正解が何位に出るかの平均逆順位 | 1/正解の順位 の平均 |
| **Precision@K** | top_Kのうち正解の割合 | 正解数 / K |

### テスト方法

```
1. テスト質問セットを用意（最低20-30件）
   - 各質問に「正解のFAQ ID」を紐づけておく
   - ユーザーが実際に聞きそうな表現で書く
   - 言い換えバリエーションも含める

2. Difyの「Retrieval Test」機能で確認
   - 各テスト質問を入力
   - top_kの結果に正解FAQ��含まれるか確認
   - 含まれない場合、何位に正解があるか記録

3. 集計
   - Hit Rate@3 = top_3に正解が含まれるクエリの割合
   - 目標: 80%以上（PoCフェーズ）、90%以上（本番フェーズ）
```

### よくある問題と対策

| 問題 | 症状 | 対策 |
|------|------|------|
| 言い換えに弱い | 「有休」で検索すると「年次有給休暇」のFAQが出ない | 言い換え・別名をFAQに追加 |
| 類似FAQの混同 | 「経費の上限」で「出張旅費」のFAQが出る | FAQ文をより具体的に。メタデータで区別 |
| チャンクが大きすぎ | 正解FAQが含まれるが、関連部分が埋もれている | チャンクサイズを縮小 |
| Embeddingの質が低い | 全般的に精度が低い | High Qualityモードに変更。Embeddingモデルを変更 |

## 2. Groundedness（根拠性）の評価

### 何���測るか
ボットの回答が、検索で拾ったFAQの情報に基づいているか。FAQに書かれていない情報を「作り出して」いないか。

### テスト方法

```
1. ボットに質問を投げ、回答を得る
2. 回答に含まれる各文（主張）について:
   a. その主張は検索されたFAQに書かれているか？
   b. FAQに書かれていない情報が含まれていないか？
   c. FAQの情報を誤って解釈していないか？

3. 判定
   - Grounded: 全ての主張がFAQに基づいている
   - Partially Grounded: 一部にFAQ外の情報が含まれる
   - Not Grounded: FAQに基づいていない情報が大部分

4. 集計
   - Groundedness Score = Grounded の割合
   - 目標: 90%以上
```

### よくある問題と対策

| 問題 | 症状 | 対策 |
|------|------|------|
| ハルシネーション | FAQにない情報を自信たっぷりに回答 | プロンプトに「ナレッジベースの情報のみ」を強調 |
| 過度な一般化 | FAQの条件を無視して一般的に回答 | プロンプトに「条件・例外を含めて」を追�� |
| 不要な補足 | FAQの回答に余計な説明を追加 | プロンプトに��回答は簡潔に」を追加 |

## 3. Completeness（網羅性）の評価

### 何を測るか
正しいFAQを拾い、根拠に沿って回答しているが、必要な条件・例外が回答に含まれているか。

### テスト方法

```
1. 各FAQの「必須要素」をリスト化
   - 結論
   - 主な条件
   - 主な例外
   - 次の行動

2. ボットの回答に各要素が含まれているか確認
   - 全て含まれている → Complete
   - 一部欠落 → Partially Complete（何が欠落しているか記録）
   - 大部分が欠落 → Incomplete

3. 集計
   - Completeness Score = Complete の割合
   - 目標: 70%以上
   - 欠落しやすい要素の傾向��分析
```

### よくある問題と対策

| 問題 | 症状 | 対策 |
|------|------|------|
| 条件の脱落 | 「正社員の場合」が回答から抜ける | FAQ自体に条件を明示的に記載 |
| 例外の脱落 | 例外ケースが回答に含まれない | FAQ自体に例外を明示的に記載 |
| 次の行動の脱落 | 「何をすればいいか」が回答にない | FAQ自体に次の行動を記載 |
| top_kが少なすぎ | 関連FAQが検索結果に含まれない | top_kを増やす（3→5） |

## 4. Resolution（解決率）の評価

### 何を測るか
最終的にユーザーが自己解決できたか。ビジネス目標に直結する指標。

### 測定方法

#### 方法A: フィードバックボタン（推奨）
```
ボットの回答の後に:
  [役に立った] [役に立たなかった]

Resolution Rate = 「役に立った」/ 全回答数
目標: 70%以上（PoCフェーズ）、80%以上（本番フェーズ）
```

#### 方法B: エスカレーション率
```
Escalation Rate = 人に回した件数 / 全問い合わせ数

目標: 30%以下（PoCフェーズ）、20%以下（本番フェ���ズ）
```

#### 方法C: フォローアップ質問率
```
Follow-up Rate = 同一セッションで追加質問があった割合

高い場合: 1回の回答で解決できていない可能性
低い場合: 1回で解決できている or 離脱している
→ エスカレーション率と組み合わせて判断
```

## 評価のワークフロー

### Phase 1: Retrieval評価（最優先）

```
1. テスト質問セットを用意（20-30件）
2. Difyの Retrieval Test で検索精度を確認
3. Hit Rate@3 が 80%未満なら:
   → ナレッジ設計（粒度、言い換え、チャンキング）を見直す
   → Retrieval が改善するまでStep 2以降は不要
```

### Phase 2: Groundedness + Completeness評価

```
1. Retrieval が80%以上になったら実施
2. ボットに質問を投げ、回答を���る
3. 各回答の根拠性と網羅性を確認
4. Groundedness が90%未満なら:
   → プロンプトの制約を強化
5. Completeness が70%未満なら:
   → FAQ自体の条件・例外記載を強化
```

### Phase 3: Resolution評価（運用開始後）

```
1. フィードバックボタンを設置
2. 2-4週間のデータを集計
3. Resolution Rate が70%未満なら:
   → 未解決の質問パターンを分析
   → FAQの追加 or 改善
```

## 症状別の原因切り分け

| 症状 | Retrieval | Groundedness | Completeness | 原因 | 対策 |
|------|:---------:|:----------:|:----------:|------|------|
| 全く的外れな回答 | NG | - | - | 検索の問題 | FAQ粒度、チャンキング、言い換えを見直す |
| 正しいが微妙 | OK | OK | NG | 網羅性の問題 | FAQに条件・例外を追加 |
| もっともらしいが嘘 | OK | NG | - | ハルシネーション | プロンプトの制約を強化 |
| 正しいが冗長 | OK | OK | OK | 出力設計 | プロンプトで「簡潔に」を指定 |
| たまに当たる | 不安定 | - | - | 言い換え不足 | 言い換え・別名を追加 |
| 正しく答えるが不満 | OK | OK | OK | FAQの範囲外 | FAQのカバレッジを拡大 or エスカレーション |
