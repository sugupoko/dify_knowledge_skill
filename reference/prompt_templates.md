# FAQ生成用プロンプトテンプレート集

## 概要

ドキュメントからボット向けFAQを生成するためのプロンプトテンプレート。
3つのパターンを使い分ける。

| パターン | 用途 | いつ使うか |
|---------|------|----------|
| A: 同時生成 | 短いドキュメントからQ&Aを一括生成 | 数ページ程度のドキュメント |
| B: 2ステップ生成 | 長い・複雑なドキュメントからQ&Aを段階的に生成 | 長いドキュメント、品質重視 |
| C: 既存FAQ変換 | 人向けFAQをボット向けに変換 | 既にFAQがある場合 |

---

## パターンA: 同時生成（短いドキュメント向け）

### プロンプト

```
以下のドキュメントを読み、このドキュメントの内容について業務担当者がよく聞きそうな質問と回答を作成してください。

## ルール
- FAQ は10-20個程度生成してください
- 「はい/いいえ」で答えられる質問ではなく、手順や理由を聞く質問にしてください
- 質問文はユーザーが実際に入力しそうな自然な話し言葉にしてください
- 1つの質問は1つのトピックに絞ってください（1FAQ1テーマ）
- 回答はドキュメントに書かれている情報のみを使ってください
- ドキュメントに該当情報がない質問は作らないでください
- 回答は「結論→条件→例外」の順序で構成してください
- 回答は2-4文で簡潔にまとめてください
- 数値（基準値、期限、金額等）は正確に記載してください

## 出力形式（タブ区切り）
カテゴリ	質問	回答（結論）	条件	例外	次の行動	参照元（章番号等）

## カテゴリ
{{categories}}
（例: 手順、基準値、トラブル対応、注意事項、申請方法）

## ドキュメント
{{document}}
```

### 使い方
1. `{{categories}}` にプロジェクトに合ったカテゴリを設定
2. `{{document}}` にドキュメント全文を貼り付け
3. 生成結果をレビューし、修正する

### 適用条件
- ドキュメントが数ページ程度（トークン数がモデルのコンテキスト長の50%以下）
- 速度重視
- 1回で済ませたい

---

## パターンB: 2ステップ生成（長い・複雑なドキュメント向け）

### Step 1: 質問を生成

```
以下のドキュメントを読み、このドキュメントの内容について業務担当者がよく聞きそうな質問を作成してください。

## ルール
- 質問は15-25個程度生成してください
- 「はい/いいえ」で答えられる質問ではなく、手順や理由を聞く質問にしてください
- 質問文はユーザーが実際に入力しそうな自然な話し言葉にしてください
- 1つの質問は1つのトピックに絞ってください（1FAQ1テーマ）
- ドキュメントに回答が書かれていない質問は作らないでください
- 以下のカテゴリに分けてください

## カテゴリ
{{categories}}

## 出力形式（タブ区切り）
カテゴリ	質問	ドキュメント内の関連箇所（章番号やページ）

## ドキュメント
{{document}}
```

### Step 2: 回答を生成

```
以下のドキュメントに基づいて、各質問に対する回答を作成してください。

## ルール
- ドキュメントに書かれている情報のみを使って回答してください
- ドキュメントに該当情報がない場合は「該当情報なし」と書いてください
- 回答は「結論→条件→例外」の順序で構成してください
- 回答の結論は1-2文で簡潔に
- 条件は「いつ当てはまるか」を明示
- 例外は「当てはまらないケース」を明示
- 数値（基準値、期限、金額等）は正確に記載してください
- 回答の末尾に、参照した箇所（章番号やページ）を記載してください
- ユーザーの次の行動（何をすればよいか）を含めてください

## 出力形式（タブ区切り）
カテゴリ	質問	回答（結論）	条件	例外	次の行動	参照元

## ドキュメント
{{document}}

## 質問一覧
{{questions_from_step1}}
```

### 使い方
1. Step 1 で質問を生成
2. 質問を人間がレビュー（不足があれば追加、不要なら削除）
3. Step 2 で回答を生成
4. 回答を人間がレビュー

### 適用条件
- ドキュメントが長い（10ページ以上）
- 品質・網羅性を重視
- 時間をかけてよい

## パターンC: 既存FAQ変換（人向けFAQ → ボット向けFAQ）

### Step 1: 変換

```
以下の既存FAQをチャットボット向けに変換してください。

## 変換ルール
- 1つの質問に複数のテーマが含まれている場合は分割してください
- 回答は「結論→条件→例外」の順序に再構成してください
- 結論は最初の1-2文で完結させてください
- 条件は「いつ当てはまるか」を明確にしてください
- 例外は「当てはまらないケース」を明確にしてください
- 「場合があります」「必要に応じて」などの曖昧な表現は、可能な限り具体的な条件に置き換えてください
- ユーザーが次に取るべき行動を明記してください
- 元のFAQに書かれていない情報は追加しないでください
- 元のFAQの情報が不足している場合は「要確認」と記載してください
- 質問文をユーザーが実際に入力しそうな自然な表現に書き換えてください

## 追加すべき情報（元のFAQにあれば抽出、なければ「要確認」）
- 条件: いつ当てはまるか
- 例外: 当てはまらないケース
- 次の行動: ユーザーが次にやること
- エスカレーション条件: 人に回す条件
- 言い換え: 略語、別名、俗称
- 参照元: 根拠となる文書名

## 出力形式（タブ区切り）
カテゴリ	質問（変換後）	回答（結論）	条件	例外	次の行動	エスカレーション条件	言い換え	参照元	変換メモ

## 変換メモの書き方
- 分割した場合: 「元FAQ #XX を分割」
- 構造変更した場合: 「結論先出しに変更」
- 情報追加した場合: 「条件を追加（要確認）」
- 変換不要の場合: 「変換なし」

## 既存FAQ
{{existing_faq}}
```

### Step 2: 不足レビュー

```
以下の変換済みFAQを確認し、不足している情報や改善点を指摘してください。

## レビュー観点
1. 1FAQ1テーマになっているか（まだ複数論点が残っていないか）
2. 結論が最初の1-2文で分かるか
3. 条件と例外が明確か（「場合があります」が残っていないか）
4. 次の行動が書いてあるか
5. エスカレーション条件が明確か
6. 言い換え・別名が十分か
7. FAQ化に向かないもの（個別判断が必要なもの）が含まれていないか

## 出力形式
FAQ番号	問題点	改善提案	優先度（高/中/低）

## 変換済みFAQ
{{converted_faq}}
```

### 使い方
1. Step 1 で既存FAQを一括変換
2. Step 2 で不足をレビュー
3. 人間が最終確認・修正

## Difyワークフローで自動化する場合

大量のドキュメントからFAQを自動生成するDifyワークフロー:

```
[Start] ドキュメントをアップロード
   ↓
[Doc Extractor] テキスト抽出
   ↓
[LLM-1] チャンク分割（章・セクション単位に分割）
   ↓
[Iteration] 各チャンクに対して繰り返し:
   │   ├── [LLM-2] 質問を生成（パターンB Step1）
   │   └── [LLM-3] 各質問に対する回答を生成（パターンB Step2）
   ↓
[Template] Excel/CSV 形式に整形
   ↓
[Answer] FAQ ファイルを出力
```

**ポイント:**
- ドキュメント全体を一度に投げるのではなく、チャンクに分けてからFAQを生成
- LLMのコンテキスト長を超えない
- チャンクごとに処理するので、大きなドキュメントでも対応可能

---

## 生成後の品質チェック

自動生成したFAQはそのまま使わず、必ず以下を確認:

| チェック項目 | 方法 |
|-------------|------|
| 事実の正確性 | 元ドキュメントと突き合わせ（参照元を確認） |
| 重複の排除 | 似た質問がないか確認。あれば統合 |
| カバレッジ | 重要なトピックが漏れていないか |
| 質問の自然さ | 実際にユーザーが聞きそうな表現か |
| 回答の長さ | 長すぎないか（結論1-2文 + 条件・例外で2-4文が目安） |
| 1FAQ1テーマ | 複数論点が混在していないか |
| 結論先出し | 結論が冒頭にあるか |
| 条件・例外 | 明確に分かれているか |

**「AIで生成、人間でレビュー」が最もコスパが良い。**

---

## パターン選択の判断表

| 条件 | 推奨パターン |
|------|-------------|
| ドキュメントが数ページ程度 | A: 同時生成（速い、1回で済む） |
| ドキュメントが長い・複雑 | B: 2ステップ（質問の網羅性を先に確保してから回答に集中） |
| 既存FAQがある | C: 変換（まず変換、不足を追加生成） |
| 既存FAQ + 追加ドキュメント | C → A or B（既存を変換してから、追加分を生成） |
| コスト・時間を重視 | A: 同時生成 |
| 品質・網羅性を重視 | B: 2ステップ |
