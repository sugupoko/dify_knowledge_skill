# ナレッジ活用手段 選定ガイド

「やりたいこと」に対して、チャットボットが最適とは限らない。
**2つの軸** で手段を選ぶ。

---

## 2軸で選ぶ

**軸1: ユーザーの状態** — 何をどこまで分かっている状態で来るか
**軸2: 情報の構造** — 答えがどこにあるか

### 選定マトリクス

| | 1箇所で完結 | 条件で分岐する | 複数箇所を組み合わせる |
|---|---|---|---|
| **キーワードが分かっている** | **検索** | **チャットフロー** | **エージェント** |
| **聞きたいことは分かっている** | **チャットボット** | **チャットフロー** | **エージェント** |
| **何を聞けばいいか分からない** | **チャットボット** | **チャットフロー** | **エージェント** |
| **聞きに来ない** | **プッシュ型** | **プッシュ型** | **プッシュ型** |

### 判断フローチャート

```
ユーザーは質問しに来る？
  ├── NO → プッシュ型（定期レポート / アラート / 通知）
  └── YES
       ↓
     答えが1箇所に書いてある？
       ├── YES
       │    ユーザーはキーワードを知っている？
       │      ├── YES → 検索型
       │      └── NO  → チャットボット
       └── NO
            ↓
          条件で答えが分岐する？
            ├── YES → チャットフロー（IF/ELSE で絞り込み）
            └── NO
                 ↓
               複数の情報源を組み合わせる必要がある？
                 ├── YES → エージェント（function calling で取りに行く）
                 └── NO  → チャットボットで十分
```

### なぜこの2軸か

**軸1（ユーザーの状態）が重要な理由:**
- キーワードを知っているユーザーに会話型を押し付けると「回りくどい」
- 何を聞けばいいか分からないユーザーに検索窓を渡しても使えない
- 聞きに来ないユーザーにはどんなUIを用意しても意味がない

**軸2（情報の構造）が重要な理由:**
- 1箇所で完結する質問にエージェントは過剰（コスト・速度の無駄）
- 条件分岐が必要な回答をチャットボットでやると誤答リスクが高い
- 複数ソースの組み合わせはチャットボットでは不可能（function calling が必要）

**ツールの機能差やアーキテクチャの話は後。まずこの2軸を見る。**

## Step 1: ユーザーの状態から手段を選ぶ

### 手段の一覧

| 分類 | 手段 | 特徴 | Dify での実現 |
|------|------|------|-------------|
| **会話型** | チャットボット | 質問→1回回答 | チャットボット + ナレッジベース |
| | チャットフロー | 分類→分岐→段階処理 | チャットフロー + Question Classifier |
| | エージェント | 自律判断→多段推論 | エージェント + ツール |
| **検索型** | セマンティック検索 | 検索→関連文書リスト表示 | チャットボット + retriever_resource 表示ON |
| | FAQ一覧+検索 | カテゴリ一覧から探す | Dify外（Webページ / Notion 等）|
| **ガイド型** | ウィザード/デシジョンツリー | 選択肢を辿って回答に到達 | チャットフロー + IF/ELSE |
| **プッシュ型** | 定期レポート | スケジュール実行→結果を配信 | ワークフロー + Schedule Trigger |
| | アラート/通知 | 条件合致で自動通知 | ワークフロー + Webhook/Slack |
| **生成型** | ドキュメント生成 | データ+テンプレート→文書作成 | ワークフロー |
| | レポート生成 | データ分析→要約→報告 | ワークフロー or エージェント |

### ユーザーの状態 × 場面 → 最適な手段

| ユーザーの状態 | 具体的な場面の例 | 最適な手段 | なぜ |
|-------------|----------------|-----------|------|
| **何を聞けばいいか分からない** | 新人が「何から始めればいいか分からない」 | チャットボット / エージェント | 曖昧に聞いても受け止めてくれる |
| **何を聞けばいいか分からない + 複雑** | 「うちの状況だとどの制度が使える？」 | エージェント | 複数ソースを横断して判断する必要がある |
| **聞きたいことは分かっているがキーワードを知らない** | 「あの書類、名前忘れたけど入社時に出すやつ」 | チャットボット | プロンプトで解釈して適切なFAQにマッチ |
| **キーワードは分かっている** | 「有給休暇の申請方法」を調べたい | セマンティック検索 | 検索窓に入れて候補から選ぶ方が速い |
| **答えが1つに決まっている** | 「営業時間は？」「返品期限は？」 | FAQ一覧 or 検索 | 一覧から探す方が速い。チャットは回りくどい |
| **質問の仕方がわからない** | 新人が「何を聞けばいいかも分からない」 | チャットボット | 曖昧に聞いても答えてくれる安心感 |
| **自分のケースがどれに当てはまるか判断したい** | 「この条件で返品できる？」「この申請はどの書式？」 | ウィザード/デシジョンツリー | 選択肢を辿る方が確実に正しい結論に到達 |
| **複数の情報を組み合わせた回答が欲しい** | 「AとBの違いを踏まえて、うちのケースはどうすべき？」 | チャットフロー or エージェント | 複数ナレッジを横断する必要がある |
| **手順の全体像を把握したい** | 「経費精算の流れを教えて」 | 検索型（手順書そのものを表示） | NN/g: 手順情報はダイレクトに欲しい。検索型で手順書を直接提示する方が速い |
| **今の作業の次のステップを知りたい** | 「申請フォームを入力した、次は？」 | チャットフロー or チャットボット（1ステップずつ案内） | CHI 2022: ステップバイステップガイドが有効。チャットフローが確実だが、プロンプト設計次第でチャットボットでも可 |
| **手順の途中で条件が分からない** | 「この場合は上長承認いる？」 | チャットフロー（条件分岐で確実に案内） | Conversation Routines (arXiv 2025): 条件分岐はフロー制御の方が信頼性が高い |
| **手順を順番に知りたい** | 「初回セットアップの手順を教えて」 | ウィザード or チャットフロー | ステップバイステップで案内する方が確実 |
| **定期的に同じ情報が欲しい** | 「毎朝、昨日の売上サマリーが欲しい」 | 定期レポート（ワークフロー） | 毎回聞くより自動配信の方が便利 |
| **異常があったら知りたい** | 「エラー率が閾値を超えたらSlackに通知」 | アラート（ワークフロー） | 人が監視するより自動検知の方が速い |
| **報告書のドラフトが欲しい** | 「月次レポートの下書きを作って」 | ドキュメント生成（ワークフロー） | チャットで頼むより、定型ワークフローの方が安定 |
| **24時間対応が必要** | 顧客向けサポート。深夜・休日も対応したい | チャットボット | 人がいない時間帯をカバーする主な理由 |
| **外部ユーザー向けの窓口** | Webサイトに埋め込むカスタマーサポート | チャットボット | 「聞ける」UIが顧客体験として強い |
| **ITリテラシーが低いユーザー** | 高齢者向け、現場作業者向け | チャットボット | 検索UIより会話の方が直感的 |

## Step 2: チャットボットが本当に最適か確認する

### チャットボットが強い場面

チャットボットが他の手段より優れるのは、以下の条件が揃うとき:

1. **ユーザーが質問の仕方を知らない**（キーワード検索では辿り着けない）
2. **曖昧な質問を受け止める必要がある**（「なんかうまくいかない」レベル）
3. **24時間対応が必要**（人がいない時間帯のカバー）
4. **外部ユーザー向け**（Webサイト埋め込み、顧客対応）
5. **対話で条件を絞り込む必要がある**（「あなたのケースはこれですか？」）

### チャットボットが弱い場面

以下の場合、チャットボットは「便利になった感じがしない」になりやすい:

| 場面 | なぜ弱いか | 代替手段 |
|------|----------|---------|
| 答えが1つに決まる定型FAQ | 一覧や検索の方が速い。チャットは回りくどい | FAQ一覧 / 検索 |
| 既にFAQ一覧を配布済み | 「同じ情報への別導線」止まりになる | 検索の方が価値が出やすい |
| 手順を正確に辿りたい | チャットだとステップが抜けるリスク | ウィザード / デシジョンツリー |
| 候補を一覧で見比べたい | チャットは1つずつ返すので比較しにくい | 検索 + 候補リスト |
| 個別判断が必要 | FAQでは答えきれず「それっぽい一般論」になる | 人へのエスカレーション |

### 判断チェックリスト

```
□ ユーザーは質問のキーワードを知っているか？
  → 知っている → 検索型の方が速い
  → 知らない → チャットボット向き

□ 答えは1つに決まるか？
  → 決まる → FAQ一覧 or 検索で十分
  → 条件による → チャットフロー or ウィザード

□ 既にFAQ一覧やマニュアルを配布しているか？
  → 配布済みで足りている → チャットボットの価値は薄い
  → 配布済みだが使われていない → チャットボットで導線改善の余地あり
  → ない → チャットボットで新規提供する価値がある

□ 24時間対応 or 外部向けか？
  → はい → チャットボットの価値が高い
  → 社内のみ・業務時間内 → 他の手段でも良い

□ 対話で絞り込みが必要か？
  → はい → チャットボット or チャットフロー
  → いいえ → 検索 or 一覧で十分
```

## Step 3: Difyの実装方式を決める

手段が「会話型」に決まった場合、以下でDifyのアプリタイプ（チャットボット/チャットフロー/エージェント）を選定する。

### 判断フローチャート

```
質問に対して1回の回答で済むか？
  ├── YES → FAQ件数は100件以下か？
  │           ├── YES → チャットボット
  │           └── NO → テーマ別に分割 → チャットフロー
  └── NO
       条件分岐や段階的な処理が必要か？
         ├── YES → チャットフロー
         └── NO
              LLMが自律的にツール選択・多段推論する必要があるか？
                ├── YES → エージェント
                └── NO → チャットフロー
```

---

## 各手段・実装方式の詳細

### 1. チャットボット

#### 向いている用途
- 定型FAQ（受付時間、手続き方法、料金）
- FAQ件数が100件以下
- 単純な質問応答（1回の検索で回答が完結）
- PoC・トライアルフェーズ
- 外部ユーザー向けの窓口

#### 具体的な場面の例

| 場面 | なぜチャットボットが良いか |
|------|----------------------|
| Webサイトのカスタマーサポート | 24時間対応。訪問者が気軽に質問できる |
| 新入社員向け社内FAQ | 何を聞けばいいか分からない新人が曖昧に質問できる |
| 製品の使い方サポート | 「うまく動かない」系の曖昧な質問を受け止められる |
| イベント・セミナーの案内ボット | 日程・場所・持ち物など定型質問に即答 |
| 社内IT ヘルプデスク | 「VPNが繋がらない」「パスワードリセットしたい」系 |

#### アーキテクチャ
```
ユーザーの質問
  → ナレッジベース検索（Semantic / Hybrid）
  → LLM が回答生成
  → 回答を返す
```

#### ナレッジの特徴
- **単一のナレッジベース** で十分
- FAQ形式（1FAQ = 1チャンク）が最も効果的
- **言い換え・別名の充実** が検索精度を左右する

#### 判断基準の数値目安
| 指標 | チャットボット向き |
|------|----------------|
| FAQ件数 | 100件以下 |
| ナレッジベース数 | 1つ |
| 条件分岐の複雑さ | 低（分岐なし or 1段階） |

---

### 2. チャットフロー

#### 向いている用途
- 部門別FAQ（質問を分類して適切なナレッジに振り分け）
- 条件によって回答が変わる質問
- 100件以上のFAQ（テーマ別に分割して精度を保つ）
- 報告書・議事録生成（データ抽出 → 整形 → 出力）
- デシジョンツリー型の絞り込み案内

#### 具体的な場面の例

| 場面 | なぜチャットフローが良いか |
|------|------------------------|
| 複数部門のFAQを統合したい | 質問を分類→適切な部門のナレッジに振り分け |
| 「この条件で申請できる？」系 | IF/ELSE で条件を絞り込んで正確に回答 |
| 見積書・報告書のドラフト生成 | データ抽出→テンプレート→整形の段階処理 |
| トラブルシュート | 症状を聞いて→原因を絞り込んで→対処法を提示 |
| 入力フォームの代わり | 対話で必要情報を収集→申請書を生成 |

#### アーキテクチャ
```
ユーザーの質問
  → Question Classifier（質問を分類）
  → 分類結果に応じてルーティング
     ├── カテゴリA → ナレッジベースA → LLM → 回答
     ├── カテゴリB → ナレッジベースB → LLM → 回答
     └── その他 → フォールバック回答
```

#### ナレッジの特徴
- **テーマ別に複数のナレッジベース** を持つ
- 各ナレッジベースの中身はチャットボットと同じFAQ形式
- **ルーティング設計** が精度の鍵
- メタデータが「ルーティングに重要」

#### 判断基準の数値目安
| 指標 | チャットフロー向き |
|------|----------------|
| FAQ件数 | 100-500件 |
| ナレッジベース数 | 2-10 |
| 条件分岐の複雑さ | 中（2-5段階の分岐） |

---

### 3. エージェント

#### 向いている用途
- 複数の情報源を横断して調査・分析
- 外部ツール（API、DB、Web検索）を使い分ける
- 多段階の推論が必要な質問
- 「まず調べて、判断して、結果を整理して報告する」タイプのタスク

#### 具体的な場面の例

| 場面 | なぜエージェントが良いか |
|------|----------------------|
| 競合調査レポート作成 | Web検索→情報収集→分析→レポート生成を自律的に |
| 障害原因の調査 | ログ、ドキュメント、過去事例を横断して原因を推定 |
| 複雑な社内規程の解釈 | 複数の規程を参照し、優先関係を考慮して判断 |
| データ分析+レポート | DB検索→集計→分析→要約を多段で実行 |
| 自動コードレビュー | コード差分→ルール参照→レビューコメント生成 |

#### アーキテクチャ
```
ユーザーの質問
  → LLM が計画を立てる
  → 必要なツール/ナレッジを自律的に選択
     ├── ナレッジ検索（必要に応じて複数回）
     ├── 外部API呼び出し
     ├── DB検索
     └── Web検索
  → 結果を統合して回答
```

#### ナレッジの特徴
- **4層のナレッジ**: FAQ + 手順書 + ポリシー + メタ情報
- **メタデータが最重要**
- 手順書はステップ単位で分割
- ポリシー・制約を知識として持たせる

#### 判断基準の数値目安
| 指標 | エージェント向き |
|------|----------------|
| FAQ件数 | 件数に依存しない |
| ナレッジベース数 | 3以上 + 外部ツール |
| 条件分岐の複雑さ | 高（事前にフロー化できない） |

---

### 4. 非会話型（ワークフロー）

#### 向いている用途
- 定期実行（レポート、集計、通知）
- Webhook / イベント駆動の自動処理
- バッチ処理（データ加工、一括変換）

#### 具体的な場面の例

| 場面 | なぜワークフローが良いか |
|------|----------------------|
| 毎朝の売上サマリーをSlackに投稿 | 毎回聞くより自動配信の方が便利 |
| サイト死活監視→アラート | 5分ごとに確認、異常時だけ通知 |
| 議事録の自動生成 | 会議録音→テキスト化→要約→保存 |
| フォーム送信→自動返信+CRM登録 | 人手不要の定型処理 |
| データの定期ETL | 外部データ取得→加工→DB登録 |

## 手段・実装方式の比較一覧

| 観点 | チャットボット | チャットフロー | エージェント | ワークフロー |
|------|:----------:|:----------:|:--------:|:--------:|
| 構築の容易さ | 簡単 | 中程度 | 難しい | 中程度 |
| メンテナンス | 低コスト | 中コスト | 高コスト | 低コスト |
| API利用料 | 最安 | 中程度 | 最も高い | 用途次第 |
| 柔軟性 | 低い | 中程度 | 高い | 中程度 |
| 予測可能性 | 高い | 高い | 低い | 高い |
| ユーザー操作 | 会話 | 会話 | 会話 | なし（自動） |

## 段階的アプローチ（推奨）

**いきなりエージェントを作らない。まず適切な手段を選び、段階的に拡張する。**

```
Phase 0: 手段の選定
  └── そもそもチャットボットが最適か確認する

Phase 1: チャットボット or 検索型
  ├── 1部門の定型FAQで PoC
  ├── 検索精度と解決率を測定
  └── 成功基準: retrieval 80%以上

Phase 2: チャットフロー
  ├── 複数部門に展開
  ├── テーマ別ナレッジベースを分割
  ├── ルーティングの精度を測定
  └── 成功基準: 分類精度 90%以上

Phase 3: エージェント or ワークフロー
  ├── 外部ツール連携
  ├── ポリシー・制約の設計
  ├── 定期実行・自動処理
  └── 成功基準: 解決率 80%以上
```

各フェーズで作ったFAQはそのまま次のフェーズで使える。
FAQの設計は共通。違うのは「持たせ方」と「使い方」。
