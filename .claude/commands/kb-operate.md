# Skill: kb-operate（運用設計）

`/kb-operate [project_path]` で実行。
展開提案が承認された後、実際に継続運用するための設計を行う。
ナレッジベースの健全性・アプリの健全性・手段の妥当性を3層で監視する。

## いつ使うか
- /kb-report で提案が承認された後
- 「これを定期的に更新・監視したい」と言われた時
- PoC から本番移行する時
- 解決率が落ちてきて「手段を見直すべきか」を判断したい時

---

## Phase 1: 運用要件の整理

### 1.1 FAQ更新サイクルの設計

```
誰が更新するか:
  ├── 各部門の担当者（コンテンツオーナー）
  ├── DX推進室などの中央管理チーム
  └── 両者の組み合わせ（部門が起票、中央が確認）

いつ更新するか:
  ├── 定期更新（月次/四半期）
  ├── トリガー更新:
  │   - 規程・制度の改定時
  │   - 未回答率が閾値（例: 10%）を超えた時
  │   - 解決率が閾値（例: 70%）を下回った時
  │   - 新しいカテゴリの問い合わせが増加した時
  └── 随時更新（現場からのフィードバック）

どう更新するか:
  ├── CSV編集 → Difyアップロード（シンプル、再現性高い）
  ├── Dify画面で直接編集（小規模な修正に適する）
  └── ドキュメント改訂 → /kb-design → /kb-build（大規模な変更）
```

### 1.2 更新フローの設計

```
標準的な更新フロー:
  1. トリガー発生（規程改定 / 問い合わせ増加 / 定期レビュー）
  2. 対象FAQの特定（何を追加・修正・削除するか）
  3. 内容の作成・修正（担当者 or /kb-design で生成）
  4. レビュー（内容の正確性を業務担当者が確認）
  5. Difyへの反映（アップロード or 画面編集）
  6. 動作確認（/kb-eval で変更箇所を重点的にテスト）
  7. 記録（spec.md の更新日・変更点を更新）
```

## Phase 2: 3層の監視設計

### 2.1 層1: ナレッジベースの健全性

```
監視指標:
  ■ FAQ鮮度
    - 最終更新日が閾値（例: 3ヶ月）を超えたFAQの数
    - 規程改定があったのに未更新のFAQがないか
    → 月次でレビュー

  ■ 未回答率
    - 「分かりません」「担当部署にお問い合わせください」で終わった割合
    - 目標: 20%以下
    → 未回答の質問パターンを収集し、FAQの追加候補に

  ■ 検索ヒット率の推移
    - Hit Rate@3 の月次推移
    - 新しい言い回しが増えていないか（言い換え追加のトリガー）
    → /kb-eval を月次で実行

アラート条件:
  - 未回答率 > 20%: FAQの追加・拡充が必要
  - 規程改定から2週間後: 対応するFAQの更新チェック
  - Hit Rate@3 が前月比-10%以上: チャンキング・言い換えの見直し
```

### 2.2 層2: アプリの健全性

```
監視指標:
  ■ レスポンスタイム
    - 平均応答時間（目標: 5秒以内）
    - タイムアウト発生率
    → Dify ログ or 外形監視

  ■ エラー率
    - API エラー率
    - Dify / LLM の障害発生状況
    → ステータスページの確認

  ■ 利用者数推移
    - 月間アクティブユーザー数
    - 利用が停滞していないか（利用者離れのシグナル）
    → 利用促進施策のトリガー

アラート条件:
  - エラー率 > 5%: Dify / LLM の障害を確認
  - レスポンスタイム > 10秒: モデル変更 / Top K 削減を検討
  - 月間利用数が前月比 -30%: 利用促進 or 手段の見直し
```

### 2.3 層3: 手段の妥当性

```
監視指標:
  ■ 解決率
    - フィードバックボタン（役に立った/立たなかった）の集計
    - 目標: 70%以上（PoC）、80%以上（本番）

  ■ エスカレーション率
    - 人に引き継いだ件数 / 全問い合わせ数
    - 目標: 30%以下（PoC）、20%以下（本番）

  ■ 条件分岐質問の比率
    - 「Aの場合は？Bの場合は？」という質問の割合
    - 30%を超えるとチャットフロー or エージェントへの移行を検討

手段変更の判断トリガー:
  ├── 解決率が目標を2ヶ月連続で下回る
  │   → FAQの拡充 or 手段変更を検討
  ├── FAQ件数が200件を超える
  │   → テーマ別ナレッジベース分割が必要
  │   → チャットボット → チャットフローへの移行を検討
  ├── 条件分岐質問が30%を超える
  │   → チャットフロー or エージェントへの移行を検討
  └── エスカレーション率が目標を2ヶ月連続で超える
      → チャットフロー or エージェントへの移行を検討
```

## Phase 3: フォールバック設計

### 3.1 Dify / LLM 停止時の対応

```
Dify が停止した場合:
  1. ステータスページを確認（status.dify.ai）
  2. 復旧まではメール / 電話での問い合わせ受付に切り替え
  3. 一時的な静的FAQページを表示（必要な場合）

LLM が利用不可の場合（APIエラー / コスト超過）:
  1. 代替モデルに切り替え（例: GPT-4o → GPT-4o-mini）
  2. Dify 設定でモデルを変更
  3. コスト超過の場合: 利用制限の緩和 or 月の途中で上限を設定し直す

回答品質が急激に低下した場合:
  1. モデルのアップデートによる変化を確認
  2. プロンプトの再調整
  3. /kb-eval で品質を再測定
```

### 3.2 FAQ更新ミス時のロールバック

```
ロールバック手順:
  1. バージョンフォルダ（v1/, v2/）で以前のCSVを保持
  2. Difyの「ドキュメント」から問題のあるFAQを削除
  3. 前バージョンのCSVを再アップロード
  4. /kb-eval で動作確認

予防策:
  - CSVをGitで管理（変更の追跡が容易）
  - 更新前に必ずバックアップ（vN/knowledge/faq_YYYYMMDD.csv）
  - 大規模更新は段階的に（一部を先にテスト）
```

## Phase 4: 手段変更の判断

### 4.1 手段バリエーション比較

現在の手段を継続するか、変更するかを以下のKPIで比較する。

```
比較対象の手段:
  - 現在の手段
  - チャットボット
  - チャットフロー
  - エージェント
```

比較テーブル（`reference/evaluation_guide.md` の手段バリエーション比較テーブルを使用）。

### 4.2 手段変更のプロセス

```
手段変更を決定した場合:
  1. 新しいバージョンフォルダ（vN+1/）を作成
  2. 新しい spec.md に変更理由と新しい手段の仕様を記録
  3. /kb-assess → /kb-design → /kb-build → /kb-eval を再実行
  → Before/After の比較は vN/ と vN+1/ の eval_report.md で確認
```

## 状態管理

### 読み込み
- バージョンフォルダ内の `.kb_state.yaml` の全セクション（assess, design, build, eval, report）を読み込む
- 評価結果（解決率、未回答率、Hit Rate）を参照して監視閾値を設定

### 書き込み
- 実行完了時にバージョンフォルダ内の `.kb_state.yaml` の `operate` セクションを書き込む
- スキーマは `reference/state_schema.md` を参照

## トラブルシューティング

### 解決率が向上しない
```
対策:
  ├── /kb-eval で原因を特定（Retrieval / Groundedness / Completeness のどこが低い？）
  ├── 未回答パターンを収集してFAQを追加
  ├── 言い換え・別名を追加してRetrieval を改善
  └── 手段変更の検討（条件分岐が多い場合はチャットフロー化）
```

### 担当者がFAQを更新してくれない
```
対策:
  ├── 更新手順を簡略化（/kb-request で更新依頼書を生成）
  ├── 更新しないことのリスクを可視化（未回答率の推移グラフ）
  ├── 更新のリマインダーを定期化（月次レビューをカレンダーに入れる）
  └── 更新作業を /kb-design で自動化（担当者が内容を確認するだけにする）
```
