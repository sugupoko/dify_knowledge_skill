# Skill: kb-design（ナレッジ設計）

`/kb-design [docs_path]` で実行。
assessの結果に基づき、ナレッジの構造を設計し、FAQ生成仕様を決める。

## いつ使うか
- /kb-assess で推奨手段とナレッジ種別が特定された後
- 「どうFAQを作るか」「どう構造化するか」を決める時

---

## Phase 1: ナレッジベース構造の設計

### 1.1 ナレッジベースの分割設計

assessで特定した推奨手段に応じて設計する。

```
■ チャットボットの場合:
  → 単一のナレッジベース
  → 全FAQを1つのKBに格納

■ 検索型の場合:
  → 単一または軽量なナレッジベース
  → FAQ形式 + retriever_resource 表示ONでソースを見せる

■ チャットフローの場合:
  → テーマ別に複数のナレッジベースに分割
  → 分割基準を決める:
    ├── 部門別（総務、IT、経理、人事）
    ├── カテゴリ別（手順、基準値、トラブル対応）
    └── 対象ユーザー別（社内、顧客）
  → ルーティング設計（Question Classifierの分類基準）

■ エージェントの場合:
  → 種類別に複数のナレッジベース
    ├── FAQ（事実系）
    ├── 手順書（手順系）
    ├── ポリシー（ルール系）
    └── メタ情報（用語集、ツール定義）
  → メタデータ設計（カテゴリ、対象業務、信頼度）

■ ワークフロー（プッシュ型）の場合:
  → ナレッジベースではなくテンプレート + データ取得ロジックで設計
  → 出力形式（Slack通知 / メール / ドキュメント）を先に決める
```

### 1.2 各ナレッジベースの仕様

各KBについて以下を定義���る:

```
ナレッジベース名: ____
目的: ____
推定項目数: ____件
ソースドキュメント: ____
チャンキング戦略: ____（reference/chunking_strategy_guide.md を参照）
メタデータ: ____
```

## Phase 2: FAQ生成仕様の決定

### 2.1 生成パターンの選択

reference/prompt_templates.md に基づいて選択する。

```
ソースの状況に応じて選択:

既存FAQあり:
  ├── ボット向けに使えるレベル → そのまま投入
  └── 人向けFAQで変換が必要 → パターンC（変換）
  
既存FAQなし:
  ├── ドキュメントが短い（数ページ） → パターンA（同時生成）
  └── ドキュメントが長い・複雑 → パターンB（2ステップ生成）

既存FAQ + 追加ドキュメント:
  → パターンC（変換）+ パターンA or B（追加生成）
```

### 2.2 FAQ生成の具体的仕��

```
■ 生成対象ドキュメント: [ファイル名のリスト]
■ 使用パターン: [A / B / C]
■ カテゴリ設定: [カテゴリのリスト]
■ 推定FAQ件数: ___件
■ 出力形式: CSV（knowledge/faq_draft.csv）
■ レビュー担当: [自分 / 依頼者 / 両方]
```

### 2.3 FAQ品質基準

reference/faq_design_guide.md のチェックリストから:

```
生成したFAQが満たすべき基準:
  □ 1FAQ1テーマ
  □ 結論先出し（最初の1-2文で回答）
  □ 条件が明示されている
  □ 例外が記載されている
  □ 次の行動が書かれている
  □ 参照元が記載されている
  □ 言い換え・別名を含む
```

## Phase 3: チャンキング戦略の決定

reference/chunking_strategy_guide.md に基づいて設計する。

### 3.1 文書タイプ別の設定

```
■ FAQ文書:
  分割単位: 1FAQ = 1チャンク
  トークン目安: 128-512
  Dify設定: Chunk Size 500, Overlap 0

■ マニュアル・手順書:
  分割単位: [全体1チャンク / セクション / ステップ]
  トークン目安: [256-1024]
  Dify設定: Chunk Size [___], Overlap [___]

■ 規程・ポリシー:
  分割単位: 条文単位
  トークン目安: 128-512
  Dify設定: Chunk Size 500, Overlap 50
```

### 3.2 前処理の必要性

```
□ Q&A形式への変換が必要（FAQ以外のドキュメント）
□ 個人情報の除去が必要
□ 表記ゆれの統一が必要
□ セクション単位への事前分割が必要
□ メタデータの付与が必要
```

## Phase 4: メタデータ設計

### 4.1 必要なメタデータフィールド

```
■ チャットボット:
  - category: カテゴリ
  - synonyms: 言い換え・別名
  - source: 参照元

■ チャットフロー（上記に加えて）:
  - routing_category: ルーティング先のカテゴリ

■ エージェント（上記に加えて）:
  - target_task: 対象業務
  - when_to_use: いつ使うべきか
  - confidence: 信頼度
  - last_updated: 最終更新日
  - valid_until: 有効期限
```

## Phase 5: プロンプト設計

### 5.1 システムプロンプト

```
推奨手段に応じたプロンプトを設計:

■ チャットボット:
  - 役割の明示
  - 回答ソースの限定（「ナレッジベースの情報に基づいて」）
  - フォールバック（「該当する情報がない場��は...」）
  - 回答の長さ制御
  - 禁止事項

■ チャットフロー:
  - 分類プロンプト（Question Classifier用）
  - 各分岐のLLMプロンプト

■ エージェント:
  - 手順（ステップ化）
  - 利用可能ツールとその用途
  - 回数制限
  - 出力形式
  - 停止条件
```
