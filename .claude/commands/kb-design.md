# Skill: kb-design（ナレッジ設計）

`/kb-design [docs_path]` で実行。
assessの結果に基づき、ナレッジの構造を設計し、FAQ生成仕様を決める。

## いつ使うか
- /kb-assess で推奨手段とナレッジ種別が特定された後
- 「どうFAQを作るか」「どう構造化するか」を決める時

---

## Phase 1: ナレッジベース構造の設計

### 1.1 ナレッジベースの分割設計

assessで特定した推奨手段に応じて設計する。

```
■ チャットボットの場合:
  → 単一のナレッジベース
  → 全FAQを1つのKBに格納

■ 検索型の場合:
  → 単一または軽量なナレッジベース
  → FAQ形式 + retriever_resource 表示ONでソースを見せる

■ チャットフローの場合:
  → テーマ別に複数のナレッジベースに分割
  → 分割基準を決める:
    ├── 部門別（総務、IT、経理、人事）
    ├── カテゴリ別（手順、基準値、トラブル対応）
    └── 対象ユーザー別（社内、顧客）
  → ルーティング設計（Question Classifierの分類基準）

■ エージェントの場合:
  → 種類別に複数のナレッジベース
    ├── FAQ（事実系）
    ├── 手順書（手順系）
    ├── ポリシー（ルール系）
    └── メタ情報（用語集、ツール定義）
  → メタデータ設計（カテゴリ、対象業務、信頼度）

■ ワークフロー（プッシュ型）の場合:
  → ナレッジベースではなくテンプレート + データ取得ロジックで設計
  → 出力形式（Slack通知 / メール / ドキュメント）を先に決める
```

### 1.2 各ナレッジベースの仕様

各KBについて以下を定義���る:

```
ナレッジベース名: ____
目的: ____
推定項目数: ____件
ソースドキュメント: ____
チャンキング戦略: ____（reference/chunking_strategy_guide.md を参照）
メタデータ: ____
```

## Phase 2: ナレッジの投入方式を決める

### 2.0 FAQ化するか、ドキュメントそのまま投入するか

**assessの結果（ドキュメント品質 + ユーザーのニーズ）で判断する。**

```
■ 判断フロー

  ユーザーのニーズは？
    ├── 「答えが欲しい」（結論を知りたい）
    │     ドキュメントの品質は？
    │       ├── 整備済み（結論が明確、1セクション1テーマ） → そのまま投入でも可
    │       └── 読む用（背景長い、論点混在） → FAQ化が必要
    │
    ├── 「該当箇所を探したい」（原文を読みたい）
    │     → ドキュメントそのまま投入（FAQ化は過剰）
    │
    └── 「手順を教えて」
          ドキュメントの品質は？
            ├── 手順書が整備済み → そのまま投入（セクション単位でチャンク）
            └── 手順が散在・不明確 → FAQ化 or 手順書の整理が必要
```

```
■ 判断基準テーブル

  | ドキュメントの状態 | ユーザーが答えを求める | ユーザーが原文を求める |
  |-----------------|:------------------:|:------------------:|
  | 構造化されている  | そのまま投入 or FAQ化  | そのまま投入          |
  | （見出し明確、1セクション1テーマ、結論あり） | | |
  | 読む用の文書     | FAQ化が必要           | ドキュメント整理が必要   |
  | （背景長い、論点混在、結論が後ろ）           | | |
```

```
■ 「そのまま投入」でよい条件（全て満たす場合）
  □ ドキュメントが構造化されている（見出し、番号、箇条書き）
  □ 1セクションが1テーマに閉じている
  □ セクション単位でチャンクしても意味が通る
  □ ユーザーは原文を読む力がある（ITリテラシーが普通以上）
  □ FAQ化のコスト（生成+レビュー）に見合う精度向上が見込めない

■ FAQ化すべき条件（いずれかに該当する場合）
  □ ドキュメントが「読む用」で、結論が埋もれている
  □ 1セクションに複数の論点が混在している
  □ ユーザーは「答え」を求めており、原文を読む時間がない
  □ 同じ質問が繰り返されており、短い定型回答の方が効率的
  □ 言い換え・別名への対応が必要（ドキュメントの表現とユーザーの言い方が違う）
```

**判定結果: [そのまま投入 / FAQ化 / 混合（一部FAQ化+一部そのまま）]**

→ そのまま投入の場合は Phase 2.1 をスキップして Phase 3 (チャンキング) へ
→ FAQ化する場合は Phase 2.1 で生成パターンを選択

### 2.1 FAQ生成パターンの選択

reference/prompt_templates.md に基づいて選択する。

```
ソースの状況に応じて選択:

既存FAQあり:
  ├── ボット向けに使えるレベル → そのまま投入
  └── 人向けFAQで変換が必要 → パターンC（変換）
  
既存FAQなし:
  ├── ドキュメントが短い（数ページ） → パターンA（同時生成）
  └── ドキュメントが長い・複雑 → パターンB（2ステップ生成）

既存FAQ + 追加ドキュメント:
  → パターンC（変換）+ パターンA or B（追加生成）
```

### 2.2 FAQ生成の具体的仕��

```
■ 生成対象ドキュメント: [ファイル名のリスト]
■ 使用パターン: [A / B / C]
■ カテゴリ設定: [カテゴリのリスト]
■ 推定FAQ件数: ___件
■ 出力形式: CSV（knowledge/faq_draft.csv）
■ レビュー担当: [自分 / 依頼者 / 両方]
```

### 2.3 FAQ品質基準

reference/faq_design_guide.md のチェックリストから:

```
生成したFAQが満たすべき基準:
  □ 1FAQ1テーマ
  □ 結論先出し（最初の1-2文で回答）
  □ 条件が明示されている
  □ 例外が記載されている
  □ 次の行動が書かれている
  □ 参照元が記載されている
  □ 言い換え・別名を含む
```

## Phase 3: チャンキング戦略の決定

reference/chunking_strategy_guide.md に基づいて設計する。

### 3.1 文書タイプ別の設定

```
■ FAQ文書:
  分割単位: 1FAQ = 1チャンク
  トークン目安: 128-512
  Dify設定: Chunk Size 500, Overlap 0

■ マニュアル・手順書:
  分割単位: [全体1チャンク / セクション / ステップ]
  トークン目安: [256-1024]
  Dify設定: Chunk Size [___], Overlap [___]

■ 規程・ポリシー:
  分割単位: 条文単位
  トークン目安: 128-512
  Dify設定: Chunk Size 500, Overlap 50
```

### 3.2 前処理の必要性

```
□ Q&A形式への変換が必要（FAQ以外のドキュメント）
□ 個人情報の除去が必要
□ 表記ゆれの統一が必要
□ セクション単位への事前分割が必要
□ メタデータの付与が必要
```

## Phase 4: メタデータ設計

### 4.1 必要なメタデータフィールド

```
■ チャットボット:
  - category: カテゴリ
  - synonyms: 言い換え・別名
  - source: 参照元

■ チャットフロー（上記に加えて）:
  - routing_category: ルーティング先のカテゴリ

■ エージェント（上記に加えて）:
  - target_task: 対象業務
  - when_to_use: いつ使うべきか
  - confidence: 信頼度
  - last_updated: 最終更新日
  - valid_until: 有効期限
```

```
  - フォールバック（「該当する情報がない場��は...」）

■ チャットフロー:
  - 分類プロンプト（Question Classifier用）
  - 各分岐のLLMプロンプト

■ エージェント:
  - 手順（ステップ化）
  - 利用可能ツールとその用途
  - 回数制限
  - 出力形式
  - 停止条件
```

## Phase 7: 出力

### 7.0 spec.md の生成

**`workspace/<project>/v1/spec.md` を生成すること。**
`reference/spec_template.md` をベースに、本プロジェクトの内容を埋める。

埋める内容:
- 情報源テーブル（R1〜: ヒアリング・ドキュメント・仮定の記録）
- 目標（主目標・副目標・対象ユーザー）
- 推奨手段と根拠（assess の結果から引用）
- ナレッジ設計テーブル（K1〜: KB構成・チャンキング・Retrieval設定）
- プロンプト設計テーブル（P1〜: 役割・禁止事項・エスカレーション条件）
- 仮定テーブル（A1〜: 未検証の仮定を全て記録）
- 受け入れ要件（利用方法・表示形式・FAQ更新方法）

spec.md を見れば「今どの条件で設計しているか」が常にわかる状態にする。
