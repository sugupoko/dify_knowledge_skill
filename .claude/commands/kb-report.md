# Skill: kb-report（展開提案書の作成）

`/kb-report [project_path]` で実行。
評価結果に基づき、ステークホルダー向けの展開提案書を作成する。

## いつ使うか
- /kb-eval で評価が一通り終わった後
- 「承認を得て展開したい」時
- 上司やクライアントに提案する時

---

## Phase 1: 結果の収集

### 1.1 全フェーズの結果を収集

```
.kb_state.yaml の全セクションを読み込み:
  - assess: 目標、ドキュメント分析、ユーザーの状態・情報の構造（2軸）、推奨手段
  - design: ナレッジ構造、FAQ仕様
  - build: 構築設定、初期テスト結果
  - eval: 評価スコア、ギャップ分析
```

### 1.2 キー数値の整理

```
■ ナレッジの規模
  - FAQ件数: ___件
  - ナレッジベース数: ___個
  - カバレッジ: ___カテゴリ

■ 評価結果
  - Retrieval: ___%
  - Groundedness: ___%
  - Completeness: ___%
  - 推定解決率: ___%

■ 改善の余地
  - 残っているギャップ: ___
  - 推奨改善アクション: ___
```

## Phase 2: 展開提案の構成

### 2.1 誰が使うか（ターゲットユーザー）

```
■ 第1フェーズのターゲット
  - ユーザー層: ____
  - 人数: ____
  - アクセス方法: ____

■ 第2フェーズ以降（拡大展開）
  - 追加ユーザー層: ____
  - 前提条件: ____
```

### 2.2 どうアクセスするか

```
アクセス方法の選択肢:
  □ Dify管理画面の共有URL
  □ Webサイトへのiframe埋め込み
  □ 社内ポータルへのウィジェット埋め込み
  □ Slack / Teams 連携
  □ API経由でカスタムUI
  □ その他: ____

推奨:
  PoCフェーズ → Dify共有URL（最速）
  本番フェーズ → 社内ポータル埋め込み or チャットツール連携
```

### 2.3 運用計画

```
■ ナレッジの更新
  - 更新頻度: [月次 / 四半期 / 必要に応じて]
  - 更新トリガー:
    ├── 制度・規程の変更
    ├── 新しいFAQの追加
    ├── ユーザーフィードバック
    └── 解決率の低下
  - 更新担当者: ____
  - 更新フロー: [担当者が直接 / 承認フロー経由]

■ 品質モニタリング
  - 確認頻度: [月次]
  - 確認指標:
    ├── 利用件数
    ├── フィードバック（Good/Bad）の比率
    ├── エスカレーション率
    └── よくある質問のトレンド変化
  - モニタリング担当者: ____

■ 改善サイクル
  - /kb-eval を定期的に実行（四半期推奨）
  - フィードバックから新しいFAQを追加
  - 古いFAQを更新・削除
```

### 2.4 コスト見積もり

```
■ 初期費用
  - ナレッジ設計・構築: ___人日
  - テスト・調整: ___人日
  - 合計: ___人日

■ 月額ランニングコスト
  - LLM API利用料:
    問い合わせ件数: ___件/月
    1件あたりのトークン数（概算）: ___tokens

    参考:

  - メンテナンス工数: ___時間/月

■ ROI（投資対効果）の概算
  - 現状の人的コスト: ___時間/月 × 時給___円 = ___円/月
  - 自動化後の削減: ___%（___円/月）
  - 投資回収期間: 約___ヶ月
```

## Phase 3: 提案書の作成

**以下のMarkdownを `reports/v1_proposal.md` に保存すること。** バージョンが進んだら `v2_proposal.md` 等に連番する。

```markdown
# {プロジェクト名} — Difyアプリ展開提案書

## エグゼクティブサマリー
- [1-3文で全体像を要約]
- [主な効果と推奨アクション]

## 現状の課題
- [具体的な問題を数字で。「月XX件の問い合わせに手動対応」等]
- [現状のコスト（時間・工数）]

## 構築したもの
- 手段: [チャットボット / 検索型 / チャットフロー / エージェント / ワークフロー]
- ナレッジ: [FAQ ___件、ナレッジベース ___個]
- 評価結果:
  | 指標 | スコア |
  |------|--------|
  | 検索精度 | ___%  |
  | 根拠性 | ___%  |
  | 網羅性 | ___%  |
  | 推定解決率 | ___% |

## 展開提案

### 提案A: [最小限の展開]（推奨）
- 対象: [ユーザー層]
- アクセス: [方法]
- コスト: [初期___人日 + 月額___円]
- 効果: [問い合わせの___%を自動化（推定）]
- 開始時期: [すぐ開始可能]

### 提案B: [拡大展開]
- 対象: [拡大後のユーザー層]
- 追加コスト: [___]
- 追加効果: [___]
- 前提条件: [提案Aの成功]

## 運用計画
- 更新頻度: [月次]
- 更新担当: [担当者/チーム]
- モニタリング: [確認指標と頻度]

## コスト見積もり
| 項目 | 金額 |
|------|------|
| 初期構築 | ___人日 |
| 月額API費用 | 約___円/月 |
| 月次メンテナンス | ___時間/月 |

## リスクと対策
| リスク | 対策 |
|--------|------|
| 検索精度が運用中に低下 | 月次の評価サイクル |
| ハルシネーション | プロンプトの制約 + エスカレーション設計 |
| ナレッジの陳腐化 | 更新トリガーの設定 |
| ユーザーの利用低下 | 開始メッセージの改善、フィードバック収集 |

## 次のステップ
1. [承認後すぐやること]
2. [1ヶ月後の評価]
3. [3ヶ月後の判断ポイント]
```

## Phase 4: 提案の優先順位付け

```
コストゼロ × 効果大 → 最優先
コスト小 × 効果大 → 次点
コスト大 × 効果大 → 要投資判断
コスト大 × 効果小 → 非推奨

※ 「段階的展開」が最もリスクが低い:
  Phase 1: 1部門でPoC（2週間）
  Phase 2: 評価して改善（2週間）
  Phase 3: 承認を得て展開（1ヶ月後）
```

## 状態管理

### 読み込み
- `.kb_state.yaml` の全セクション（assess, design, build, eval）を読み込む
- 全フェーズの履歴を参照して提案書を作成

### 書き込み
- 実行完了時に `.kb_state.yaml` の `report` セクションを書き込む
- 出力ファイルパス、提案リスト、コスト見積もりを記録
- スキーマは `reference/state_schema.md` を参照
