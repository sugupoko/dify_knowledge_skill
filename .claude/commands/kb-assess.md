# Skill: kb-assess（アセスメント）

`/kb-assess [docs_path]` で実行。

**全スキルの中で最も重要。** ここで手段の判断やナレッジ種別の見立てを間違えると、後の全てが無駄になる。

## いつ使うか
- クライアントやチームから「これでボット作って」とドキュメントを渡された時
- 何を作るべきか自体がまだ曖昧な時

---

## Phase 1: ドキュメントの理解（何があるか）

### 1.1 ドキュメントの中身を把握する
```
- 全ファイルを開いて構造を確認（ページ数、シート数、形式）
- 文書の種類を分類: FAQ / マニュアル / 規程 / Q&Aログ / スプレッドシート / その他
- 言語を確認（日本語/英語/混在）
- データ品質を確認（最終更新日、欠損、表記ゆれ）
```

### 1.2 既存FAQがある場合の品質チェック
```
□ 1FAQ1テーマになっているか？ → 複数論点が混在していないか
□ 結論が先に来ているか？ → 背景説明から始まっていないか
□ 条件・例外が明示されているか？ → 「場合があります」で逃げていないか
□ 言い換え・別名があるか？ → ユーザーの聞き方とFAQの表現が近いか
□ 参照元が記載されているか？ → 根拠が追えるか

→ これらが「NO」なら、人向けFAQでありボット向けへの変換が必要
→ reference/faq_design_guide.md の基準で判定
```

## Phase 2: 目標の理解（何を作るか）

### 2.1 依頼者へのヒアリング項目

```
■ 目的（What）
  「何を実現したいですか？」
    ├── 問い合わせ対応の自動化
    ├── ドキュメント検索の効率化
    ├── 業務手順のガイド
    └── その他

■ 対象ユーザー（Who）
  「誰が使いますか？」
    ├── 社内の従業員 → 専門用語OK
    ├── 外部の顧客 → 平易な表現が必要
    └── 特定部署 → 業務特化の設計

■ 規模（How much）
  「どれくらいの量ですか？」
    ├── FAQ件数（既存 + 想定追加）
    ├── 問い合わせ件数/月
    └── 対象ユーザー数

■ 制約（Must not）
  「ボットが答えてはいけない範囲は？」
    ├── 個人情報
    ├── 法的アドバイス
    ├── 人事評価・給与
    └── その他

■ 成功基準（How to judge）
  「成功とは何ですか？」
    ├── 問い合わせの___%を自動化
    ├── 回答時間を短縮
    └── 担当者の負荷を軽減
```

### 2.2 手段の判定

reference/app_type_selection_guide.md に基づいて判定する。
**2つの軸** で決める。

```
■ 軸1: ユーザーの状態（何をどこまで分かっている状態で来るか）

  □ 何を聞けばいいか分からない → 会話型（チャットボット / エージェント）
  □ 聞きたいことは分かっているがキーワードを知らない → チャットボット
  □ キーワードは分かっている → 検索型
  □ 自分のケースがどれに当たるか分からない → チャットフロー（絞り込み）
  □ そもそも質問しない（聞きに来ない） → プッシュ型（ワークフロー）

■ 軸2: 情報の構造（答えがどこにあるか）

  □ 1箇所で完結する → チャットボット or 検索
  □ 条件で答えが分岐する → チャットフロー
  □ 複数の情報源を組み合わせる必要がある → エージェント（function calling）
```

```
■ 選定マトリクス

             | 1箇所で完結  | 条件で分岐   | 複数箇所を組み合わせ |
  -----------|------------|------------|------------------|
  KW分かる    | 検索        | チャットフロー | エージェント       |
  聞きたいこと | チャットボット | チャットフロー | エージェント       |
  何も分からない| チャットボット | チャットフロー | エージェント       |
  聞きに来ない | プッシュ型    | プッシュ型    | プッシュ型         |
```

```
■ 判断フローチャート

  ユーザーは質問しに来る？
    ├── NO → プッシュ型（ワークフロー）
    └── YES
         答えが1箇所に書いてある？
           ├── YES → ユーザーはキーワードを知っている？
           │           ├── YES → 検索型
           │           └── NO  → チャットボット
           └── NO
                条件で答えが分岐する？
                  ├── YES → チャットフロー
                  └── NO
                       複数の情報源を組み合わせる必要がある？
                         ├── YES → エージェント
                         └── NO  → チャットボットで十分
```

### 2.3 ナレッジ登録方式の判定

推奨手段が決まったら、ナレッジの登録方式も決める。
ユーザーが判断に迷わないよう、assessの段階で方針を出す。

```
■ 推奨手段 → 登録方式の対応

  検索型        → 元文書のみ（FAQ化は過剰。ユーザーは原文を読みたい）
  チャットボット → FAQのみ（短い回答で完結。元文書は参照リンクで十分）
  チャットフロー → FAQのみ（KB別に分割。ルーティングで振り分ける）
  エージェント   → FAQ + 元文書（自律的に深掘りする。元文書も検索対象に）
  ワークフロー   → テンプレート + データ取得ロジック（ナレッジベースは使わない場合も）

■ ドキュメントの整備状況による補正

  元文書が整備済み（構造化、見出し明確、1セクション1テーマ）:
    → チャットボットでも「元文書のみ」で十分な場合がある
    → FAQ化の工数を省ける

  元文書が読む用（背景長い、論点混在）:
    → 検索型でもFAQ化した方が検索精度が上がる場合がある
```

```
■ 判定テーブル

  | 推奨手段 | 元文書が整備済み | 元文書が読む用 |
  |---------|:-------------:|:----------:|
  | 検索型 | 元文書のみ | 元文書 + 要約FAQ |
  | チャットボット | 元文書のみ or FAQ | FAQ |
  | チャットフロー | FAQ (KB別) | FAQ (KB別) |
  | エージェント | FAQ + 元文書 | FAQ + 元文書 |
  | ワークフロー | テンプレート | テンプレート |
```

### 2.3 必要なナレッジ種別の特定

reference/knowledge_structure_guide.md に基づいて判定する。

```
3層のどこまで必要か:

層1: FAQ（直接回答） → 常に必要
  推定件数: ___件
  ソース: 既存FAQ変換 ___件 + ドキュメントから生成 ___件

層2: 支援文書 → 手順書やルールが必要な場合
  □ 手順書: [必要 / 不要]
  □ 規程・ポリシー: [必要 / 不要]
  □ 用語集: [必要 / 不要]

層3: 運用情報 → エスカレーションやメンテナンスが必要な場合
  □ エスカレーションルール: [必要 / 不要]
  □ 更新管理: [必要 / 不要]
  □ 禁止事項: [必要 / 不要]
```

## Phase 3: 仮定と不足情報の整理

### 3.1 仮定を明示する

**これが最も重要。** ドキュメントやヒアリングで確認できなかった部分は仮の値を置くが、それを明示する。

```
仮定のリスト:
  - 「既存FAQ85件のうち、60件はそのまま使えると仮定」
  - 「問い合わせは月100件程度と仮定」
  - 「ユーザーのITリテラシーは普通と仮定」

なぜ大事か:
  仮定が間違っていれば設計も間違う。
  仮定を書くと、担当者が「いやそれは違う、実際はXX」と教えてくれる。
```

### 3.2 不足情報のリスト化
```
追加で必要な情報:
  - [ ] よくある問い合わせのトップ20（FAQの優先順位付けに必要）
  - [ ] エスカレーション基準の明文化（ボットの動作制御に必要）
  - [ ] 最新の規程・マニュアル（古い情報で回答しないために必要）
```

## 状態管理

### 読み込み
- `.kb_state.yaml` が存在すれば読み込み、前回の assess 結果を確認する
- 追加ドキュメントが来た場合は既存の assess を参照して差分を分析する

### 書き込み
- 実行完了時に `.kb_state.yaml` の `assess` セクションを書き込む
- スキーマは `reference/state_schema.md` を参照
