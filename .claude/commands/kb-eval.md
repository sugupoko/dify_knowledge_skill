# Skill: kb-eval（評価）

`/kb-eval [project_path]` で実行。
構築したDifyアプリの品質を、検索精度・根拠性・網羅性・解決率の4軸で評価する。

## いつ使うか
- /kb-build でDifyアプリを構築した後
- 「品質は大丈夫か？」を確認する時
- 改善サイクルの中で再評価する時

---

## Phase 1: テスト質問セットの準備

### 1.1 テスト質問の設計

```
最低20-30件のテスト質問を用意する。

質問の種類:
  ■ 正常系（15-20件）
    - FAQ に直接対応する質問
    - 各カテゴリから均等に
    - 質問: [テスト質問]
    - 正解FAQ ID: [対応するFAQ]
    - 期待する回答のキーワード: [含まれるべき情報]
  
  ■ 言い換え系（5-8件）
    - 同じFAQに対して別の表現で聞く
    - 略語、俗称、カジュアルな表現
    
  ■ 境界系（3-5件）
    - FAQにギリギリ該当する/しない質問
    - 条件分岐が発生する質問
    
  ■ 範囲外系（3-5件）
    - FAQに該当しない質問
    - 答えてはいけない質問
    - → フォールバックが正しく動くか確認
```

### 1.2 テスト質問セットのフォーマット

```yaml
test_queries:
  - id: "T001"
    query: "経費精算の提出期限はいつですか？"
    type: "normal"
    expected_faq_id: "FAQ-001"
    expected_keywords: ["毎月末日", "前営業日"]
    expected_conditions: ["全社員共通"]
    expected_exceptions: ["海外出張"]
  
  - id: "T002"
    query: "経費の締め切り教えて"
    type: "paraphrase"
    expected_faq_id: "FAQ-001"
    expected_keywords: ["毎月末日"]
  
  - id: "T003"
    query: "社長の給料いくら？"
    type: "out_of_scope"
    expected_behavior: "fallback"
    expected_response_contains: ["お答えできません", "担当部署"]
```

## Phase 2: Retrieval（検索精度）評価

### 2.1 Dify Retrieval Testの実行

```
Dify管理画面 → ナレッジ → {KB名} → Retrieval Test

各テスト質問を入力し、以下を記録:
  - top_3の検索結果（FAQ IDとスコア）
  - 正解FAQがtop_3に含まれるか（Hit/Miss）
  - 正解FAQの順位
```

### 2.2 集計

```
Hit Rate@3 = top_3に正解が含まれるクエリ数 / 全クエリ数
MRR = 1/正解の順位 の平均

目標:
  - Hit Rate@3 ≥ 0.80（PoCフェーズ）
  - Hit Rate@3 ≥ 0.90（本番フェーズ）
```

### 2.3 Missの分析

```
Missだった質問について:
  1. 何が代わりにヒットしたか
  2. なぜ正解FAQが拾えなかったか
     ├── 言い換えが不足 → FAQ/メタデータに言い換えを追加
     ├── チャンクが大きすぎ → 分割
     ├── 類似FAQと混同 → FAQ文を差別化
     └── Embeddingの限界 → hybrid search検討
  3. 改善アクションを決定
```

## Phase 3: Groundedness（根拠性）評価

### 3.1 回答の根拠チェック

```
正常系テスト質問に対するボットの回答を取得し、以下を確認:

各回答について:
  1. 回答に含まれる各文（主張）を列挙
  2. 各主張が検索されたFAQに書かれているか判定:
     - Grounded: FAQの情報に基づいている
     - Not Grounded: FAQに書かれていない情報が含まれている
     - Partially Grounded: 一部はFAQに基づいているが、追加情報がある
```

### 3.2 集計

```
Groundedness Score = Grounded の回答数 / 全回答数

目標: ≥ 0.90

Not Groundedの場合の原因:
  ├── LLMのハルシネーション → プロンプト強化
  ├── FAQの情報が曖昧でLLMが補完 → FAQ自体を修正
  └── 複数FAQの情報を混ぜている → top_kを減らす or プロンプト調整
```

## Phase 4: Completeness（網羅性）評価

### 4.1 必須要素のチェック

```
各テスト質問の回答について、以下が含まれているか確認:

  □ 結論（回答の核心部分）
  □ 主な条件（いつ当てはまるか）
  □ 主な例外（当てはまらないケース）
  □ 次の行動（ユーザーがやること）

判定:
  - Complete: 全ての必須要素が含まれている
  - Partially Complete: 一部が欠落
  - Incomplete: 大部分が欠落
```

### 4.2 集計

```
Completeness Score = Complete の回答数 / 全回答数

目標: ≥ 0.70

欠落パターンの分析:
  □ 条件の脱落が多い → FAQ自体に条件を明示的に記載
  □ 例外の脱落が多い → FAQ自体に例外を明示的に記載
  □ 次の行動の脱落が多い → FAQ自体に次の行動を記載
  □ 全体的に薄い → top_kを増やす、プロンプトに「条件・例外を含めて」追加
```

## Phase 5: フォールバック・エスカレーション評価

### 5.1 範囲外質問のテスト

```
範囲外テスト質問に対して:
  □ 「分からない」と正しく返すか
  □ 推測で回答していないか
  □ エスカレーション先を案内するか
  □ 答えてはいけない質問に答えていないか
```

### 5.2 集計

```
Fallback Accuracy = 正しくフォールバックした数 / 範囲外質問数

目標: ≥ 0.90

問題があった場合:
  ├── 無理に回答している → Score Threshold を上げる
  ├── フォールバック文言が不適切 → プロンプトを修正
  └── エスカレーション先が案内されない → プロンプトにエスカレーション情報を追加
```

## Phase 6: ギャップ分析と改善提案

### 6.1 ギャップの整理

```
評価結果から、以下のギャップを特定:

■ Retrieval ギャップ
  - Hit Rate が低い質問パターン: ____
  - 原因: ____
  - 改善策: ____

■ Groundedness ギャップ
  - ハルシネーションが発生した質問: ____
  - 原因: ____
  - 改善策: ____

■ Completeness ギャップ
  - 条件・例外が抜けた質問: ____
  - 原因: ____
  - 改善策: ____

■ フォールバック ギャップ
  - 正しくフォールバックしなかった質問: ____
  - 原因: ____
  - 改善策: ____
```

### 6.2 改善の優先順位

```
優先度の判断基準:
  1. Retrieval の改善（検索が外れていれば他は全て無意味）
  2. Groundedness の改善（ハルシネーションは信頼を損なう）
  3. Fallback の改善（答えてはいけない質問に答えるのは危険）
  4. Completeness の改善（条件・例外の充実）
```

## Phase 6.5: 不可能性の判断

評価結果から、**現在の手段では目標達成が困難かどうか**を判断する。
「改善すれば届く」のか「手段そのものを変えるべき」のかを明確にする。

### 不可能性の判断基準

```
以下の条件に該当する場合、手段変更を検討すること:

■ 検索精度の限界
  □ Hit Rate@3 が80%に届かない、かつ
    - チャンキング改善・言い換え追加をしても改善しない
    → チャットボットのFAQアプローチ自体が限界の可能性
    → チャットフロー（テーマ別分割+ルーティング）への移行を検討

■ FAQ件数の爆発
  □ FAQ件数が200件を超える見込み
    → 単一ナレッジベースの管理・精度維持が困難
    → テーマ別ナレッジベースの分割が必要（チャットフロー化）

■ 条件分岐の多さ
  □ 条件分岐質問（「Aの場合は？Bの場合は？」）が全体の30%を超える
    → チャットボットで答えるより条件分岐フローを設計した方が正確
    → チャットフロー or エージェントへの移行が必須

■ ハルシネーションの多発
  □ Groundedness が80%を下回り、プロンプト改善でも解決しない
    → ナレッジベースの内容が曖昧・複雑すぎる可能性
    → ドキュメント自体の整理が先か、エージェント化を検討

■ 自己解決率の構造的限界
  □ 手順・プロセスが複数ステップにわたる問い合わせが多い
    → FAQでは対応困難（1問1答の形式が合わない）
    → チャットフロー（ステップ案内型）or エージェントを検討
```

### impossibility_findings の記録

```yaml
impossibility_findings:
  - finding: "Hit Rate@3 が70%で、3回のチャンキング改善後も80%に届かない"
    implication: "チャットボット単体では精度目標を達成できない"
    recommendation: "チャットフローによるテーマ別分割を検討"
    evidence: "IT系の略語・専門用語への対応が根本的に困難"
  - finding: "FAQ件数が推定200件を超えている"
    implication: "単一KBでは管理・精度維持が困難"
    recommendation: "部門別に3〜4のKBに分割（チャットフロー化）"
    evidence: "総務/IT/経理で話題の性質が全く異なる"
```
