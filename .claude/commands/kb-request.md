# Skill: kb-request（確認依頼書の生成）

`/kb-request` で実行。
設計・構築・評価の途中で「業務担当者に確認したい」となった時に、確認依頼書を自動生成する。

## いつ使うか
- /kb-assess で仮定が多い時、業務担当者に確認が必要な時
- /kb-design で FAQ の内容や運用ルールを確認したい時
- /kb-eval で評価結果を業務担当者に共有して妥当性を確認したい時
- 途中でエスカレーション基準や更新フローが不明確になった時

---

## Phase 1: 確認すべき内容の分類

### 1.1 4パターンの確認形式

```
パターン1: FAQ内容の確認
  「この回答で正しいですか？」形式
  → 生成したFAQの内容・条件・例外が実務と一致するか確認
  → 特に: 例外ケース、条件分岐、エスカレーション先

パターン2: エスカレーション基準の確認
  「この条件で人に回してよいですか？」形式
  → どの質問を自動回答し、どの質問を人に回すかの境界を確認
  → 特に: クレーム・法的問題・例外対応・VIP顧客

パターン3: 仮定値の確認
  「現在の仮定一覧です。修正があれば教えてください」形式
  → spec.md の仮定テーブルを見せて訂正を促す
  → 特に: confidence: low の仮定を優先的に確認

パターン4: 運用確認
  「FAQ更新は誰が、どの頻度で行いますか？」形式
  → 誰が・いつ・どうやってFAQを更新するかを確認
  → 特に: 更新担当者、更新トリガー、承認フロー
```

### 1.2 確認の書き方のルール

**「教えてください」ではなく「こう理解していますが合っていますか？」の形式で書く。**
業務担当者は白紙の質問には答えにくいが、具体的な仮説を見せると「いやそれは違う」と訂正してくれる。

```
✗ 悪い例（答えにくい）:
  「エスカレーション基準を教えてください」
  「FAQ更新のフローを教えてください」

○ 良い例（答えやすい）:
  「クレームや法的問題は自動回答せず、担当者に引き継ぐと理解しています。
   以下のどれに近いですか？
     A) その通り（クレーム・法的問題・医療相談は全て人に回す）
     B) クレームでも一次回答はボットでよい
     C) 法的問題のみ必ず人に回し、クレームはケースバイケース」

○ 良い例（仮定を見せて訂正を促す）:
  「FAQ件数を150件と仮定して設計しました。
   既存FAQが85件あり、マニュアルからさらに65件追加できると想定しています。
   この認識で合っていますか？
     A) だいたい合っている
     B) 既存FAQは使えないものが多い（実際に使えるのは___件くらい）
     C) もっと多いカテゴリがある（___に関する質問が多い）」
```

## Phase 2: 確認依頼書の生成

### 2.1 パターン1: FAQ内容の確認

```
確認対象のFAQを選ぶ:
  - 業務ルールが複雑で例外が多いもの
  - 条件分岐が発生するもの
  - ヒアリングで情報が不足していたもの
  - confidence: low の仮定に依存するもの

確認形式:
  「Q: [質問]
   A: [生成した回答]

   この回答について:
   □ 正しい
   □ 条件が不足（追記: ＿＿＿）
   □ 例外が不足（追記: ＿＿＿）
   □ 内容が誤っている（正しくは: ＿＿＿）」
```

### 2.2 パターン2: エスカレーション基準の確認

```
エスカレーション基準の確認形式:
  「以下のケースは、ボットではなく人に引き継ぐと設計しています。
   追加・変更があれば教えてください。

   現在の設計:
   □ クレーム・苦情（感情的なメッセージが含まれる場合）
   □ 法的問題（法律・訴訟・契約に関する質問）
   □ 例外申請（規定外の処理を求める場合）
   □ 個人情報・機密情報に関する質問
   □ ボットが「分からない」と回答した場合

   追加すべきケース: ＿＿＿
   削除すべきケース: ＿＿＿」
```

### 2.3 パターン3: 仮定値の確認

```
spec.md の仮定テーブルから confidence: low / medium の仮定を抽出して提示:

  「以下の仮定で設計を進めています。
   「これは違う」があれば教えてください。

   | 仮定 | 現在の値 | この値が変わると |
   |------|---------|----------------|
   | [仮定1] | [値] | [影響] |
   | [仮定2] | [値] | [影響] |

   修正がある場合: ＿＿＿」
```

### 2.4 パターン4: 運用確認

```
運用に関する確認形式:
  「FAQ更新の運用について確認させてください。
   現在の理解:
     更新担当: [担当者/チーム]
     更新頻度: [月次/随時]
     更新トリガー: [規程改定/問い合わせ増加/etc]
     更新手順: [CSV編集→アップロード/Dify画面操作]
   
   以下のどれに近いですか？
     A) この通り
     B) 更新担当が違う（実際は: ＿＿＿）
     C) 更新頻度が違う（実際は: ＿＿＿）
     D) その他（具体的に: ＿＿＿）」
```

## 状態管理

### 読み込み
- `.kb_state.yaml` の全セクションを読み込む
- 現在の仮定、不足情報、評価ギャップを参照
- spec.md が存在する場合は仮定テーブルと QA チェックリストを参照

### 書き込み
- 依頼書を生成した事実を `.kb_state.yaml` に記録（該当フェーズのセクション内）
- スキーマは `reference/state_schema.md` を参照

---

## トラブルシューティング

### 何を確認すべきか分からない
```
対策:
  ├── spec.md の仮定テーブル（confidence: low の項目）を見る
  ├── .kb_state.yaml の missing_info フィールドを見る
  ├── eval の gaps フィールドを見る（評価後の場合）
  └── 「この質問が来た時にどう回答すべきか分からない」という項目を抽出
```

### 担当者からの返答が遅い・少ない
```
対策:
  ├── 質問数を絞る（1回の依頼は最大5問まで）
  ├── 選択式にする（記述式は答えにくい）
  ├── 影響を明示する（「これが分からないと○○が進められません」）
  └── 仮定で進める宣言をする（「○日までに回答がなければAで進めます」）
```
