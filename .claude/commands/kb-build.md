# Skill: kb-build（Difyアプリ構築）

`/kb-build [project_path]` で実行。
designの設計仕様に基づき、Difyアプリを実際に構築する（APIまたはUIガイド）。

## いつ使うか
- /kb-design でナレッジ設計が完了した後
- 「実際にDifyで動くものを作る」時

---

## Phase 1: 構築方法の選択

### 1.1 API vs UIガイド

```
■ API構築（推奨: FAQ件数が多い場合）
  - reference/dify_api_reference.md を使用
  - ナレッジベース作成 → FAQアップロード → アプリ設定をスクリプトで実行
  - 利点: 大量のFAQを一括投入可能、再現性が高い

■ UIガイド（推奨: 初回構築や少数のFAQ）
  - Dify管理画面でのステップバイステップガイドを生成
  - 利点: 非技術者でも実行可能、画面を見ながら確認できる

■ ハイブリッド（推奨: 多くの場合）
  - ナレッジベース作成とFAQアップロードはAPI
  - アプリ設定とプロンプト調整はUI
```

---

## Phase 1.5: FAQ / ナレッジデータの自動生成

**spec.md と design_report.md の仕様に従って、実際のFAQデータを生成する。**
ドキュメントそのまま投入の場合はこのPhaseをスキップする。

### 1.5.1 生成パターンの確認

spec.md の FAQ生成仕様を確認する:
- パターンA（同時生成）/ B（2ステップ）/ C（既存FAQ変換）のどれか
- 対象ドキュメント
- カテゴリ設定
- 推定件数

### 1.5.2 FAQ の自動生成

reference/prompt_templates.md のプロンプトを使い、対象ドキュメントからFAQを生成する。

```
■ 出力先: v{N}/knowledge/faq_draft.csv（TSV形式）

■ 出力フォーマット:
  No | カテゴリ | 質問 | 回答 | 条件/例外 | 次の行動 | エスカレーション | 参照元 | 言い換えキーワード

■ パターン別の実行手順:

  パターンA（同時生成 — ドキュメントが短い場合）:
    1. 対象ドキュメントを読み込む
    2. prompt_templates.md のパターンAプロンプトを適用
    3. 出力を faq_draft.csv に保存

  パターンB（2ステップ — ドキュメントが長い場合）:
    1. 対象ドキュメントをチャンク（章・セクション単位）に分割
    2. 各チャンクからStep 1: 質問を生成
    3. 各質問に対して Step 2: 回答を生成
    4. 結合して faq_draft.csv に保存

  パターンC（既存FAQ変換）:
    1. 既存FAQを読み込む
    2. prompt_templates.md のパターンCプロンプトで変換
    3. 不足テーマがあれば対象ドキュメントからパターンA/Bで追加生成
    4. 結合して faq_draft.csv に保存
```

### 1.5.3 生成結果の品質チェック

reference/faq_design_guide.md のチェックリストで確認:
```
□ 1FAQ1テーマになっているか
□ 質問文がユーザーの言い方に近いか
□ 回答の最初の1-2文で結論が分かるか
□ 条件と例外が分かれているか
□ 次の行動が書かれているか
□ エスカレーション条件があるか
□ 参照元が記載されているか
□ 言い換え・別名があるか
□ 推定件数と実際の生成件数が大きく乖離していないか
```

### 1.5.4 レビュー待ち

```
■ 生成完了後、以下をユーザーに提示:
  - faq_draft.csv の件数とカテゴリ別内訳
  - 品質チェック結果
  - 「内容を確認してください。修正があれば教えてください」

■ レビュー後の対応:
  - 修正指示があれば faq_draft.csv を更新
  - 問題なければ Phase 2 に進む
```

## Phase 2: ナレッジベースの構築

### 2.1 ナレッジベースの作成

designで定義した各KBを作成する。

```
API の場合:
  POST /datasets
  {
    "name": "{KB名}",
    "description": "{説明}",
    "indexing_technique": "high_quality"
  }

UIガイドの場合:
  1. Dify管理画面 → ナレッジ → 作成
  2. 名前: {KB名}
  3. 説明: {説明}
  4. インデクシングモード: High Quality
```

### 2.2 FAQデータのアップロード

knowledge/faq_draft.csv からFAQを投入する。

```
■ 方法1: テキストで1件ずつ（API）
  → reference/dify_api_reference.md のバッチ処理パターンを使用
  → 1FAQ = 1ドキュメントとしてアップロード

■ 方法2: CSVファイルで一括（UI）
  1. Dify管理画面 → ナレッジ → {KB名} → ドキュメント追加
  2. CSVファイルをアップロード
  3. チャンキング設定を確認
  4. インデクシングを開始

■ 方法3: Q&A形式で投入（UI）
  1. Dify管理画面 → ナレッジ → {KB名} → ドキュメント追加
  2. 「Notion Sync」ではなく「テキストファイル」を選択
  3. Markdownまたはテキストファイルをアップロード
  4. セグメンテーション設定でQ&Aペアが分割されないよう注意
```

### 2.3 チャンキング設定の確認

designで決めたチャンキング設定を適用する。

```
Dify UIでの設定:
  1. ドキュメント追加時の「Process Settings」
  2. Mode: Custom
  3. Separator: [designで決めた区切り文字]
  4. Max Tokens: [designで決めたサイズ]
  5. Overlap: [designで決めたオーバーラップ]
```

### 2.4 インデクシングの確認

```
アップロード後:
  1. 各ドキュメントの indexing_status が "completed" になるまで待つ
  2. Segment数が想定通りか確認
  3. サンプルのセグメントを開いて内容を確認
  → Q&Aペアが途中で切れていないか
  → メタデータが正しく付与されているか
```

## Phase 3: アプリ / ワークフローの構築

### 3.1 チャットボット / 検索型の場合

```
UIでの構築手順:
  1. Dify管理画面 → アプリ → 作成 → チャットボット
  2. アプリ名を設定
  3. モデルを選択（designで決めたモデル）
  4. システムプロンプトを設定（designで設計したプロンプト）
  5. ナレッジベースを接続
     - 「コンテキスト」→ 「追加」→ 作成したKBを選択
     - Retrieval Mode: Semantic Search（推奨）
     - Top K: 3（推奨。後で調整）
     - Score Threshold: 0.5（推奨。後で調整）
  6. 開始メッセージを設定
     - 「何ができるか」+ サンプル質問を含める
  7. 保存してプレビュー
```

### 3.2 チャットフローの場合

```
UIでの構築手順:
  1. Dify管理画面 → アプリ → 作成 → チャットフロー
  2. フローを設計:
     [Start]
       ↓
     [Question Classifier] ← designの分類カテゴリを設定
       ├── カテゴリA → [Knowledge Retrieval: KBA] → [LLM] → [Answer]
       ├── カテゴリB → [Knowledge Retrieval: KBB] → [LLM] → [Answer]
       └── その他 → [LLM: フォールバック回答] → [Answer]
  
  3. 各ノードを設定:
     - Question Classifier: designの分類プロンプトを設定
     - Knowledge Retrieval: 対応するKBを接続
     - LLM: 各分岐のプロンプトを設定
     - Answer: 出力形式を設定
  
  4. 保存してプレビュー
```

### 3.3 エージェントの場合

```
UIでの構築手順:
  1. Dify管理画面 → アプリ → 作成 → エージェント
  2. モデルを選択（推論能力が高いモデルを推奨）
  3. システムプロンプトを設定（designで設計したプロンプト）
     - 手順をステップ化
     - 利用可能ツールとその用途を明記
     - 回数制限・停止条件を明記
  4. ツールを追加:
     - ナレッジベース検索（作成した各KBを接続）
     - 必要に応じて外部ツール
  5. 保存してプレビュー
```

### 3.4 ワークフロー（プッシュ型）の場合

```
UIでの構築手順:
  1. Dify管理画面 → アプリ → 作成 → ワークフロー
  2. トリガーを設定:
     - Schedule Trigger: 定期実行（毎朝、毎週等）
     - Webhook: 外部イベント起動
  3. フローを設計:
     [Trigger]
       ↓
     [HTTP Request / DB Lookup] ← データ取得
       ↓
     [LLM] ← 要約・生成・判断
       ↓
     [出力: Slack通知 / メール / ドキュメント保存]
  4. 出力フォーマット（テンプレート）を設定
  5. テスト実行 → タイムアウトや失敗パターンを確認
```

## Phase 4: 初期テスト

### 4.1 基本動作確認

```
以下のテストを実施:

1. 正常系: よくある質問を3-5件投げて、正しく回答するか
2. 言い換え: 同じ質問を別の表現で聞いて、同じ回答が返るか
3. フォールバック: ナレッジにない質問を投げて、適切に「分からない」と返すか
4. エスカレーション: エスカレーション条件に該当する質問を投げて、正しく人に回すか
5. 禁止事項: 答えてはいけない質問を投げて、回答を控えるか
```

### 4.2 問題があった場合の調整

```
回答が不正確:
  → Retrieval Test で検索結果を確認
  → 正しいFAQが拾えていない → チャンキング/言い換えを調整
  → 正しいFAQが拾えている → プロンプトを調整

回答が冗長:
  → プロンプトに「簡潔に」を追加
  → Top Kを減らす

フォールバックが動かない:
  → Score Threshold を上げる
  → プロンプトに「該当情報がない場合は」を強調
```

## Phase 4.5: QA チェック（設計↔実装の突合）

構築完了後、**spec.md の設計と Dify の実際の設定が一致しているか**を確認する。
このチェックをスキップすると「設計通りに作ったつもりが実は違った」という問題が後から発覚する。

### QA チェックリスト

```
□ QA1: プロンプトの一致
  - spec.md のプロンプト設計テーブル（P1〜P6）の全項目が
    Difyの実際のシステムプロンプトに含まれているか
  - チェック方法: spec.md を開き、Dify画面の「システムプロンプト」と比較
  - 不一致の場合: Difyのプロンプトを spec.md に合わせて修正

□ QA2: チャンキング設定の一致
  - spec.md のナレッジ設計テーブル（K2〜K4）のチャンキング設定が
    Difyの実際のドキュメント設定と一致しているか
  - チェック方法: Difyの「ナレッジ」→「ドキュメント」→「設定」で確認
  - 不一致の場合: 設定を修正するか spec.md を更新して理由を記録

□ QA3: 禁止事項の反映
  - spec.md の禁止事項（P5: 答えてはいけない内容）が
    プロンプトに全て含まれているか
  - チェック方法: spec.md の P5 を1項目ずつプロンプトで検索
  - 不一致の場合: プロンプトに禁止事項を追記

□ QA4: エスカレーション条件の反映
  - spec.md のエスカレーション条件（P6）が
    プロンプトに明示的に記載されているか
  - チェック方法: フォールバックフローをテストして確認
  - 不一致の場合: プロンプトにエスカレーション条件を追記

□ QA5: 仮定の検証状況
  - spec.md の仮定テーブル（A1〜）のうち、
    confidence: low の仮定が実際のデータで検証済みか
  - チェック方法: spec.md の仮定テーブルで「検証状況: 未検証」の項目を確認
  - 未検証の場合: /kb-request で確認依頼書を生成し、業務担当者に確認
```

### QA チェック結果の記録

|---|------------|------|---------|
| QA1 | プロンプトの一致 | [OK/NG/一部NG] | [不一致の箇所があれば記録] |
| QA2 | チャンキング設定の一致 | [OK/NG] | |

## 状態管理

### 読み込み
- `.kb_state.yaml` の `design` セクションを読み込む
- ナレッジベース構造、FAQ生成仕様、チャンキング設定を引き継ぐ

### 書き込み
- 実行完了時に `.kb_state.yaml` の `build` セクションを書き込む
- スキーマは `reference/state_schema.md` を参照
