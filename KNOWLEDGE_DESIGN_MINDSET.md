# ナレッジ設計スペシャリストの思考回路 — 抽象化と判断基準

## この文書の目的

ナレッジ設計スペシャリストがDifyアプリのナレッジを設計するときの思考パターンを言語化し、「この状況ではこう考える」という判断基準を紐づける。LLMや非専門家がナレッジ設計に取り組む際のガイドとして使う。

## ① 人向けFAQ ≠ ボット向けFAQ（変換の思考）

### 思考

> 「人向けFAQをそのままRAGに入れるな。検索されて切り出されて使われることを前提に再構成しろ。」

人向けFAQは「読めばわかる」でよい。ボット向けFAQは「拾われても崩れない」が必要。

### 人向けとボット向けの違い

| | 人向けFAQ | ボット向けFAQ |
|---|----------|-------------|
| 前提 | 人が読んで理解する | 検索されて切り出されて使われる |
| 長さ | 多少長くてもOK | 短い知識ユニットが有利 |
| 構造 | 背景から説明しても読める | 結論が先でないと使いにくい |
| 曖昧さ | 人が文脈を補える | そのまま誤答につながる |
| 複数論点 | 1項目に混ぜても読める | 検索ノイズになる |

### 変換が必要なパターン

| 元のFAQの特徴 | 問題 | 変換方法 |
|-------------|------|---------|
| 背景説明から始まる | 結論が埋もれる | 結論を冒頭に移動 |
| 1項目に複数論点 | 検索時にノイズ | テーマ別に分割 |
| 「場合があります」が多い | 判断軸がない | 条件を明示 |
| 「詳細は別紙」で終わる | 自己完結しない | 必要な情報を含めるか、参照先を具体化 |
| 専門用語のみ | ユーザーの言い方と乖離 | 言い換え・別名を追加 |

### 判断フロー

```
既存のFAQがあるか？
  ├── ある → ボット向けに変換が必要か判定
  │     ├── 1FAQ1テーマ？ → NO → 分割
  │     ├── 結論先出し？ → NO → 構造変更
  │     ├── 条件明示？ → NO → 条件・例外を追加
  │     └── 言い換えあり？ → NO → 別名・俗称を追加
  └── ない → ドキュメントからFAQを新規生成
        → reference/prompt_templates.md を使用
```

## ② 1テーマ = 1知識ユニット（粒度の設計）

### 思考

> 「1つのFAQ項目に複数のテーマを詰め込むな。検索で拾われた時にノイズになる。1テーマ1ユニットが鉄則。」

### 粒度の判断基準

| 粒度 | いつ使うか | 例 |
|------|----------|------|
| **細かすぎ** | 1文だけで文脈が失われる | 「受付は9時からです」→ 何の受付？ |
| **適切** | 1つの質問に完結して答えられる | 「経費精算の提出期限はいつ？」→ 回答 + 条件 + 例外 |
| **粗すぎ** | 複数テーマが混在 | 「経費精算について」→ 提出期限 + 上限額 + 承認フロー + 例外 |

### 分割の判断

```
1項目に含まれるテーマ数は？
  ├── 1つ → そのまま
  ├── 2-3つ → 分割を検討（それぞれ独立して答えられるか？）
  └── 4つ以上 → 必ず分割
        │
        分割後の各項目は自己完結するか？
        ├── する → 分割OK
        └── しない → 最低限の文脈を各項目に含める
```

### 悪い分割の例

| 悪い分割 | なぜ悪いか |
|---------|----------|
| 「Q: 経費精算について」を「Q: 提出期限は？」「Q: 上限額は？」に分割するが条件が各項目になし | 「正社員の場合」「契約社員の場合」が落ちる |
| 手順書を1ステップずつ分割して前後関係を失う | 「前のステップが完了している前提で」が伝わらない |

## ③ 結論先出し、条件は後（構造の設計）

### 思考

> 「ボットの回答は結論から始めろ。背景や前置きから入ると、LLMが本題に入る前にトークンを使い切る。」

### 構造テンプレート

```
■ 良い構造:
  結論（1-2文）
  条件（いつ当てはまるか）
  例外（当てはまらないケース）
  次の行動（ユーザーがやること）

■ 悪い構造:
  背景説明
  前提条件
  詳細な説明
  結論（最後にやっと出てくる）
```

### Lost in the Middle 問題との関係

Liu et al. (2024) が示した通り、LLMは長い入力の中間部分の情報を活用しにくい。FAQ項目が長く、結論が中間や末尾にあると:
- 検索で正しいFAQを拾えても、結論がLLMに拾われない
- 前置き部分で回答が構成され、本題が薄くなる

**対策: 結論を冒頭に置く。これだけで回答精度が上がる。**

## ④ 3層で持つ: FAQ + 支援文書 + 運用情報（深さ）

### 思考

> 「FAQだけでは足りない。手順書、ルール、エスカレーション情報がないと、ボットは"それっぽいが使えない"回答を返す。」

### 3層の構造

```
層1: FAQ（直接回答層）
  ├── ユーザーの質問に直接答える短い知識ユニット
  ├── 1FAQ1テーマ、結論先出し
  └── 検索の第一候補

層2: 支援文書（詳細・手順層）
  ├── 手順書・マニュアル
  ├── 規程・ポリシー
  ├── 入力検証ルール
  ├── 用語集・別名辞書
  └── FAQだけでは答えきれない時の詳細情報

層3: 運用情報（メタ層）
  ├── エスカレーションルール（誰に、何を添えて渡すか）
  ├── 更新責任者・最終更新日
  ├── 有効期間
  └── ボットが答えてはいけない範囲の定義
```

### 手段別の必要度

| 層 | チャットボット | 検索型 | チャットフロー | エージェント | ワークフロー |
|----|:----------:|:----:|:----------:|:--------:|:--------:|
| 層1: FAQ | 必須 | 必須 | 必須 | 必須 | 不要 |
| 層2: 支援文書 | リンク程度 | 参考文書 | 必要 | 必須（ステップ分割） | テンプレートとして必須 |
| 層3: 運用情報 | FAQ内に含める | 不要 | フロー設計に含む | 自律判断の制約として必須 | 更新管理が必須 |

### 判断フロー

```
現在のナレッジは何層あるか？
  ├── 層1のみ（FAQだけ）
  │     → 層2（手順書・ルール）の追加を検討
  │     → 層3（エスカレーション）の追加を検討
  ├── 層1+層2
  │     → 運用情報の追加を検討（更新責任、有効期間）
  └── 3層揃っている
        → 各層の品質チェックへ
```

## ⑤ メタデータの重要度は手段で変わる（適応）

### 思考

> 「チャットボットなら言い換えが命。エージェントならメタデータが命。選んだ手段に合わせてナレッジの"持たせ方"を変えろ。」

### 手段別のメタデータ設計

| メタデータ | チャットボット | 検索型 | チャットフロー | エージェント |
|-----------|:----------:|:----:|:----------:|:--------:|
| カテゴリ | あると良い | あると良い | ルーティングに必須 | 検索判断に必須 |
| 対象業務 | 不要 | 不要 | あると良い | 必須 |
| いつ使うべきか | 不要 | 不要 | 不要 | 必須 |
| 信頼度 | 不要 | 不要 | 不要 | 推奨 |
| 最終更新日 | あると良い | あると良い | あると良い | 必須 |
| 言い換え・別名 | 必須 | 必須 | 必須 | 必須 |
| 参照元（文書名・ページ） | 推奨 | 必須 | 推奨 | 必須 |

### なぜエージェントではメタデータが重要か

チャットボット・検索型ではユーザーの質問で直接検索する。検索精度 = 言い換えの吸収力。

エージェントでは「検索するかどうか」「どのナレッジを検索するか」をLLMが自律判断する。
その判断材料がメタデータ。メタデータがないと:
- 不要な検索を繰り返す
- 適切なナレッジベースを選べない
- 古い情報と新しい情報の区別がつかない

## ⑥ 評価 = 検索 + 根拠性 + 網羅性（測定）

### 思考

> 「回答が自然に見えるかだけで評価するな。正しいFAQを拾えたか、根拠に沿っているか、必要な情報を落としていないかを分けて測れ。」

### 評価の4軸

| 軸 | 見るもの | 測り方 |
|----|---------|--------|
| **Retrieval（検索精度）** | 正しいFAQを拾えたか | テスト質問でtop_kに正解が入るか |
| **Groundedness（根拠性）** | 拾ったFAQに基づいて答えているか | 回答に含まれる情報がFAQにあるか |
| **Completeness（網羅性）** | 必要な条件・例外を落としていないか | FAQの条件・例外が回答に反映されているか |
| **Resolution（解決率）** | ユーザーが自己解決できたか | フォローアップ質問の有無、エスカレーション率 |

### 症状別の原因切り分け

| 症状 | Retrieval | Groundedness | 原因 |
|------|:---------:|:----------:|------|
| 正しいFAQを拾っているのに回答が微妙 | OK | NG | プロンプト設計の問題 |
| 違うFAQを拾う | NG | - | FAQ粒度 or チャンキングの問題 |
| 正しいが条件が抜けている | OK | OK | Completenessの問題（FAQに条件が書かれていない） |
| たまに当たるが安定しない | 不安定 | - | 言い換え吸収が弱い |
| 正しく答えるが満足されない | OK | OK | FAQで解けない問い合わせをFAQで解こうとしている |

### 判断フロー

```
ボットの回答品質に問題がある
  ↓
まずRetrievalを確認
  ├── 正しいFAQを拾えていない → ナレッジ設計（粒度、言い換え、チャンキング）を見直す
  └── 正しいFAQを拾えている
        ↓
        Groundednessを確認
        ├── FAQにない情報を言っている → プロンプトの制約を強化
        └── FAQに沿っている
              ↓
              Completenessを確認
              ├── 条件・例外が抜けている → FAQ自体に条件・例外を追加
              └── 全て含まれている
                    ↓
                    Resolutionを確認
                    ├── エスカレーション率が高い → FAQのカバレッジを拡大
                    └── 解決率が高い → 品質良好
```

## 思考回路の適用ガイド — ケース別

### ケースA: 人向けFAQを渡されて「これでボット作って」と言われた

```
推奨順序: ① → ② → ③ → ④

  ① 人向けFAQをそのまま使えるか判定
  ② 1FAQ1テーマに分割
  ③ 結論先出しに構造変更
  ④ 支援文書・運用情報の不足を確認
```

### ケースB: マニュアルだけ渡されてFAQがない

```
推奨順序: ② → ③ → ① → ④

  ② まずドキュメントからFAQを生成（テーマ単位に分割）
  ③ 生成したFAQを結論先出しに整形
  ① 人間にレビューしてもらい変換品質を確認
  ④ 支援文書として元マニュアルも保持
```

### ケースC: ボットを作ったが「イマイチ」と言われた

```
推奨順序: ⑥ → ① → ② → ③

  ⑥ まず評価（Retrieval / Groundedness / Completeness のどこが弱いか）
  ① 人向けFAQのまま入れていないか確認
  ② 粒度は適切か（1FAQ1テーマか）
  ③ 構造は適切か（結論先出しか）
```

### ケースD: 手段を変更・拡張したい（チャットフロー、エージェント、ワークフロー等）

```
推奨順序: ⑤ → ④ → ② → ⑥

  ⑤ 変更後の手段に合ったメタデータ設計
  ④ 3層（FAQ + 支援文書 + 運用情報）の充実
  ② ナレッジベースのテーマ別分割設計
  ⑥ 拡張後の評価
```

### ケースE: ステークホルダーに提案する

```
推奨順序: ⑥ → ④ → ⑤ → ①

  ⑥ 評価数値で効果を見せる（検索精度XX%、解決率XX%）
  ④ 「FAQだけでなく支援文書も必要」の根拠を示す
  ⑤ 手段別のコスト・効果比較
  ① Before/After（人向けFAQ vs ボット向けFAQ）の具体例
```

## まとめ: 6つの思考回路の関係

```
ドキュメントを受け取る
  │
  ① 人向け？ボット向け？（変換の要否を判断）
  │
  ② テーマ単位に分割（粒度の設計）
  │
  ③ 結論先出し、条件は後（構造の設計）
  │
  ④ 3層で持つ（FAQ + 支援文書 + 運用情報）
  │
  ⑤ 選んだ手段に合わせてメタデータ設計（適応）
  │
  ⑥ 評価して改善（検索 + 根拠性 + 網羅性）
      │
      └─→ ①に戻って改善サイクル
```

この6つは独立ではなく、**評価結果を見て設計を直す**サイクルが回る。「検索精度が低い→粒度を見直す→再評価」のサイクルの回転数と質で結果が決まる。
